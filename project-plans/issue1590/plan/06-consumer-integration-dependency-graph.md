**Full Text**: Update direct non-core consumers to import moved storage APIs from `@vybestack/llxprt-code-storage`.

- GIVEN: non-core packages directly import moved storage APIs from core
- WHEN: those imports are updated
- THEN: packages depend on and use the new storage package directly for storage primitives

**Why This Matters**: Direct consumers prove storage is a real shared package and not an isolated copy hidden behind core.

### REQ-NOCYCLE-001: No Circular Dependencies

**Full Text**: Ensure no package dependency cycle is introduced between storage, core, mcp, providers, cli, and a2a-server.
**Behavior**:

- GIVEN: core and consumers now depend on storage
- WHEN: package dependencies and imports are analyzed

**Why This Matters**: The issue requires a foundational package that higher-level packages can build on.

### Split-Import Guidance for Mixed Imports

When a file imports both moved storage symbols and non-moved core symbols from `@vybestack/llxprt-code-core`, the import must be split into two separate statements. Apply this exact pattern:

**Before** (single import from core):
```typescript
import { Storage, ConfigConstructor, debugLogger } from '@vybestack/llxprt-code-core';
```

**After** (split: storage symbols from storage, core symbols from core):
```typescript
import { Storage } from '@vybestack/llxprt-code-storage';
import { ConfigConstructor, debugLogger } from '@vybestack/llxprt-code-core';
```

**Rules**:
1. Moved symbols (`Storage`, `SecureStore`, `FileDiscoveryService`, `ConversationFileWriter`, `LLXPRT_DIR`, `SESSION_FILE_PREFIX`, etc. — see full list in specification.md "Tier 1: Stable Root API") MUST be imported from `@vybestack/llxprt-code-storage`.
2. Non-moved symbols (`debugLogger`, `DebugLogger`, `ConfigConstructor`, `coreEvents`, `FatalConfigError`, `createToolRegistry`, etc.) MUST remain imported from `@vybestack/llxprt-code-core`.
3. Do NOT create a third import from core deep paths for moved symbols — use storage root or storage deep paths instead.
4. If a file imports only moved symbols, the entire import statement moves to storage (no split needed).
5. If a file imports only non-moved symbols, no change is needed.

**Type-only imports**: Apply the same split. `import type { ConversationRecord } from '@vybestack/llxprt-code-core'` becomes `import type { ConversationRecord } from '@vybestack/llxprt-code-storage'`.

**Dynamic imports**: `const { Storage } = await import('@vybestack/llxprt-code-core')` becomes `const { Storage } = await import('@vybestack/llxprt-code-storage')`.

**vi.mock split pattern**: When a test mocks both moved and non-moved symbols, split into two mocks:
```typescript
// Before:
vi.mock('@vybestack/llxprt-code-core', async (importOriginal) => ({
  ...(await importOriginal<typeof import('@vybestack/llxprt-code-core')>()),
  Storage: { getGlobalLlxprtDir: vi.fn() },
  createToolRegistry: vi.fn(),
}));

// After:
vi.mock('@vybestack/llxprt-code-storage', async (importOriginal) => ({
  ...(await importOriginal<typeof import('@vybestack/llxprt-code-storage')>()),
  Storage: { getGlobalLlxprtDir: vi.fn() },
}));
vi.mock('@vybestack/llxprt-code-core', async (importOriginal) => ({
  ...(await importOriginal<typeof import('@vybestack/llxprt-code-core')>()),
  createToolRegistry: vi.fn(),
}));
```

**For files that mock or dynamically import moved symbols from core**, the rewrite must update both `vi.mock(...)` paths and any `vi.importActual(...)` calls that reference core for moved symbols.
