## P06 Remediation (post-P06a-failure fixes)

### Issue 1: Core ProfileManager consumers still imported core-relative profileManager
Updated four core consumers to import `ProfileManager` from `@vybestack/llxprt-code-settings`:
- `packages/core/src/config/subagentManager.ts`
- `packages/core/src/config/toolRegistryFactory.ts`
- `packages/core/src/tools/task.ts`
- `packages/core/src/core/subagentOrchestrator.ts`

Also updated `packages/core/src/config/configBaseCore.ts` (type import) and `packages/core/src/index.ts` (re-export) to use settings-owned `ProfileManager`.

### Issue 2: Old core settingsServiceInstance.ts bridged singleton + runtime context
Removed runtime-context imports (`createProviderRuntimeContext`, `setActiveProviderRuntimeContext`, `clearActiveProviderRuntimeContext`) from `settingsServiceInstance.ts`. Core's old `registerSettingsService()` and `resetSettingsService()` now only manage the local module variable; they do NOT create/set/clear provider runtime context. `settingsRuntimeAdapter.ts` remains the sole production bridge.

Updated `providerRuntimeContext.test.ts` to assert P06 single-owner semantics: core `resetSettingsService()` does NOT clear provider runtime context.

### Issue 3: P06a graph check included devDependencies in cycle DFS
Updated `06a-core-integration-stub-verification.md` to use production-only dependencies for cycle DFS (aligns with P05a semantics), while still checking both dependencies and devDependencies for settings forbidden deps.

### Issue 4: configBaseCore.ts comment text matched `from.*memoryTool` scan
Replaced "Decoupled from memoryTool to eliminate configBaseCore → tools dependency" with "Decoupled from memory tool to eliminate configBaseCore dependency on tools" — removes the false-positive `from.*memoryTool` match while preserving behavioral documentation.

### Issue 5: Root barrel `index.ts` re-exported both settings singleton lifecycle AND runtime-context helpers (P06a bridge scan violation)
`packages/core/src/index.ts` re-exported `getSettingsService`, `registerSettingsService`, `resetSettingsService` (settings singleton lifecycle) alongside `createProviderRuntimeContext`, `setActiveProviderRuntimeContext`, `clearActiveProviderRuntimeContext`, `peekActiveProviderRuntimeContext` (runtime-context helpers). This violates the P06a single-owner bridge rule: only `settingsRuntimeAdapter.ts` may bridge/expose both sides in production.

Fix: removed the three settings singleton lifecycle exports from the core root barrel. Updated 6 external consumers that imported these symbols from `@vybestack/llxprt-code-core`:

Production consumers (2):
- `packages/cli/src/auth/provider-usage-info.ts` — `getSettingsService` now imported from `@vybestack/llxprt-code-settings`
- `packages/cli/src/providers/providerManagerInstance.ts` — `getSettingsService` now imported from `@vybestack/llxprt-code-settings`

Test consumers (4):
- `packages/cli/src/auth/oauth-manager.issue1317.spec.ts` — `SettingsService`, `resetSettingsService`, `registerSettingsService` now imported from `@vybestack/llxprt-code-settings`
- `packages/cli/src/auth/oauth-manager.spec.ts` — same
- `packages/cli/src/auth/oauth-manager.token-reuse.spec.ts` — same
- `packages/cli/src/config/config.loadMemory.test.ts` — `resetSettingsService` now imported from `@vybestack/llxprt-code-settings`

Note: `SettingsService` class was moved to settings package import for tests that pass it to settings-package `registerSettingsService()` to avoid cross-package type mismatch (core vs settings `SettingsService` are structurally identical but TypeScript treats them as distinct types).
