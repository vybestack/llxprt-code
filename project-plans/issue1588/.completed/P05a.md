## Source Review Evidence

### SettingsService set/get and provider-settings behavior

- `packages/settings/src/settings/SettingsService.ts`
  - `set()` stores non-dotted keys in `settings.global` and dotted keys through `setNestedValue()`.
  - `get()` returns global values or nested values via dot notation.
  - `setProviderSetting()` initializes a provider map as needed, stores provider-local settings, and emits `provider-change`.
  - `getProviderSettings()` returns the provider settings object or `{}` for absent/invalid provider state.
  - `exportForProfile()` exports default provider, provider settings, and tools settings.
  - `importFromProfile()` applies default provider, provider settings, and tools settings.

### ProfileManager save/load behavior with real filesystem JSON semantics

- `packages/settings/src/profiles/ProfileManager.ts`
  - `saveProfile()` creates the profile directory recursively and writes pretty-printed JSON using `fs.writeFile(..., JSON.stringify(profile, null, 2), 'utf8')`.
  - `loadProfile()` reads JSON from `<profilesDir>/<profileName>.json`, parses it, validates required fields/version, validates load-balancer references, maps missing files to `Profile '<name>' not found`, and maps parse failures to corrupted-profile errors.
  - `listProfiles()`, `deleteProfile()`, and `profileExists()` use real filesystem semantics.
  - `save()` and `load()` bridge settings-like services through real profile JSON conversion without importing core settings types.

### Settings singleton register/get/reset behavior

- `packages/settings/src/settings/settingsServiceInstance.ts`
  - `registerSettingsService()` stores the singleton.
  - `getSettingsService()` returns the registered singleton or throws when missing.
  - `resetSettingsService()` clears the registered service if present and resets the package-owned singleton to `null`.
  - No core runtime-context imports are present.

### Settings registry behavior and compression strategy values

- `packages/settings/src/settings/settingsRegistry.ts`
  - Registry is settings-owned and does not import core compression types.
  - Compression strategies are inlined as literal values: `middle-out`, `top-down-truncation`, `one-shot`, `high-density`.
  - `compression.strategy` uses these local literal values, with default `middle-out` and enum validation through registry behavior.

### Storage path behavior and own LLXPRT_DIR constant

- `packages/settings/src/storage/Storage.ts`
  - Defines its own `export const LLXPRT_DIR = '.llxprt'`.
  - Path helpers preserve global settings, installation id, provider accounts, OAuth, commands, skills, policies, temp, project settings, history, checkpoints, and extensions path behavior.
  - No core/tools import is used for `LLXPRT_DIR` or path behavior.
