## What Upstream Does

Upstream allows extensions to contribute agent definitions via `llxprt-extension.json` (originally `gemini-extension.json`) manifests that become available to the runtime alongside built-in and user-defined agents. The upstream intent:

1. Extension manifests declare agent (→ subagent in LLxprt) definitions.
2. Contributed agents are discovered during extension loading (`loadExtension()`).
3. The runtime makes them available for delegation and listing.
4. The CLI/UI surfaces expose these extension-provided agents coherently alongside user-defined ones.

## Why REIMPLEMENT in LLxprt

1. LLxprt does **not** use upstream `/agents`, `AgentRegistry`, or markdown-frontmatter agent architecture.
2. LLxprt has `/subagent` command, `SubagentManager`, JSON-backed subagent definitions (`~/.llxprt/subagents/*.json`), and `task()` delegation semantics.
3. The correct LLxprt adaptation is **extension-provided subagents** flowing through `SubagentManager` semantics.
4. The existing `GeminiCLIExtension` interface already carries `skills` and `mcpServers` from extensions — adding `subagents` follows the same pattern.
5. There should be no existing extension-subagent integration (confirm via preflight checks).
6. This must preserve LLxprt naming: `/subagent`, `SubagentManager`, `task()`.

## Design Decisions

### D1: Extend SubagentConfig with optional `source` provenance

Add an optional `source` field to `SubagentConfig` in `packages/core/src/config/types.ts`:

```typescript
/** Where this subagent came from. Omitted or 'user' = user-created JSON. */
source?: 'user' | 'extension';
/** Extension name that contributed this subagent, if source === 'extension'. */
sourceExtension?: string;
```

This is backward-compatible — existing JSON files on disk lack this field and default to `'user'` behavior. Extension-contributed subagents will have `source: 'extension'`.

### D2: Extend GeminiCLIExtension with `subagents` array

Add to `GeminiCLIExtension` in `packages/core/src/config/config.ts`:

```typescript
subagents?: Array<{
  name: string;
  profile: string;
  systemPrompt: string;
}>;
```

This mirrors how `skills` are already carried on `GeminiCLIExtension`.

### D3: Extend ExtensionConfig with `subagents` manifest field

Add to `ExtensionConfig` in `packages/cli/src/config/extension.ts`:

```typescript
subagents?: Array<{
  name: string;
  profile: string;
  systemPrompt: string;
}>;
```

This allows `llxprt-extension.json` to declare subagents.

### D4: Adapt SubagentManager to merge extension-contributed subagents

Add a method to `SubagentManager` to register extension-contributed subagents in memory (not on disk). The manager will maintain an in-memory map for extension subagents alongside its existing disk-based storage.

**Conflict policy when two extensions define the same subagent name:** Last-registered wins (registration order follows `ExtensionLoader.getExtensions()` array order). A `console.warn()` is emitted at registration time naming both extensions. This keeps behavior deterministic and observable.

`listSubagents()` returns the union of user and extension subagents, deduplicated by name (user-first precedence per D5), sorted alphabetically. The sort ensures deterministic listing regardless of registration or discovery order.

`loadSubagent()` checks disk (user) first, then falls back to the in-memory extension map — see D5 for precedence rationale.

### D5: Precedence — user-owned subagents win

If a user-defined `~/.llxprt/subagents/foo.json` exists and an extension also contributes a subagent named `foo`, the **user-defined one wins** for `loadSubagent()` resolution. This matches the principle that user config always overrides extension-provided defaults.

**`listSubagents()` semantics:** Returns the alphabetically sorted union of all subagent names. When a name exists in both user disk storage and the extension map, it appears **once** (the user version takes precedence for subsequent `loadSubagent()` calls). The listing is deterministic: sorted alphabetically, no duplicates.

**`loadSubagent()` resolution order:** (1) disk (user) → (2) in-memory extension map → (3) throw not-found. This means if both exist, the user disk version is always returned.

### D6: Extension subagents are read-only through /subagent UI — with user-override mutability rule

Extension-contributed subagents cannot be edited or deleted through `/subagent edit` or `/subagent delete` **unless a user disk override exists for that name**. Specifically:

- **Extension-only (no user disk file):** `/subagent edit` and `/subagent delete` reject with an error: "Cannot edit/delete extension-provided subagent 'foo'. It is managed by extension 'bar'."
- **User disk override exists for same name:** `/subagent edit` and `/subagent delete` operate on the **user disk file**. Deleting the user override causes the extension version to become visible again (the extension version is never modified or removed by user actions).
- **User-only (no extension):** Normal behavior, no change.

The UI should indicate provenance when showing them (e.g., `[ext: my-extension]` label, or `[ext: my-extension, user override]` when both exist).

### Step 4: Adapt SubagentManager for extension subagents

In `packages/core/src/config/subagentManager.ts`, add:

**4a.** A private in-memory map for extension-contributed subagents:
```typescript
private extensionSubagents: Map<string, SubagentConfig> = new Map();
```

**4b.** A public method to register extension subagents (with cross-extension conflict warning):
```typescript
registerExtensionSubagents(
  extensionName: string,
  subagents: Array<{ name: string; profile: string; systemPrompt: string }>,
): void {
  for (const sa of subagents) {
    // Warn if another extension already registered this name (cross-extension conflict)
    const existing = this.extensionSubagents.get(sa.name);
    if (existing && existing.sourceExtension !== extensionName) {
      console.warn(
        `Extension '${extensionName}' overrides subagent '${sa.name}' previously registered by extension '${existing.sourceExtension}'. Last-registered wins.`,
      );
    }
    const now = new Date().toISOString();
    this.extensionSubagents.set(sa.name, {
      name: sa.name,
      profile: sa.profile,
      systemPrompt: sa.systemPrompt,
      createdAt: now,
      updatedAt: now,
      source: 'extension',
      sourceExtension: extensionName,
    });
  }
}
```

**4c.** Public methods to clear extension subagents (for reload/refresh):
```typescript
/** Clear ALL extension subagents (used during full re-initialization). */
clearExtensionSubagents(): void {
  this.extensionSubagents.clear();
}

/** Remove subagents contributed by a specific extension (used when an extension is unloaded). */
removeExtensionSubagents(extensionName: string): void {
  for (const [name, config] of this.extensionSubagents) {
    if (config.sourceExtension === extensionName) {
      this.extensionSubagents.delete(name);
    }
  }
}
```

**4d.** Modify `listSubagents()` to include extension subagents (user-first precedence):
```typescript
async listSubagents(): Promise<string[]> {
  // Get disk-backed user subagents (existing logic)
  const userSubagents = await this._listDiskSubagents();
  // Merge extension subagents (only names not already in user list)
  const extensionNames = [...this.extensionSubagents.keys()].filter(
    (name) => !userSubagents.includes(name),
  );
  return [...userSubagents, ...extensionNames].sort();
}
```

Extract the current `listSubagents()` body into a private `_listDiskSubagents()` method.

**4e.** Modify `loadSubagent()` to check disk first, then fall back to extension subagents:
```typescript
async loadSubagent(name: string): Promise<SubagentConfig> {
  // Try disk-backed (user) subagent first
  try {
    const config = await this._loadDiskSubagent(name);
    return { ...config, source: 'user' };
  } catch {
    // Fall through to extension lookup
  }
  // Check extension-contributed subagents
  const extSubagent = this.extensionSubagents.get(name);
  if (extSubagent) {
    return extSubagent;
  }
  throw new Error(ERROR_MESSAGES.SUBAGENT_NOT_FOUND.replace('{name}', name));
}
```

Extract the current `loadSubagent()` body into a private `_loadDiskSubagent()` method.

**4f.** Modify `subagentExists()` to also check extension subagents:

async subagentExists(name: string): Promise<boolean> {

Extract the current `subagentExists()` body into a private `_diskSubagentExists()` method.

async deleteSubagent(name: string): Promise<boolean> {

    const diskExists = await this._diskSubagentExists(name);

## Execution Risks

1. **Precedence correctness.** User-owned subagents must always win over extension-provided ones in `loadSubagent()`. `listSubagents()` must not contain duplicates. Test both explicitly.
2. **Mutability guard with user-override edge case.** Extension-only subagents must not be editable/deletable. But if a user disk override exists for the same name, edit/delete must operate on the user disk file. Deleting the user override must cause the extension version to resurface. Test all three cases: extension-only, user-override-of-extension, user-only.
3. **Cross-extension conflict.** When two extensions define the same subagent name, last-registered wins and a `console.warn()` is emitted. Test that the warning fires and the last-registered version is returned.
4. **Profile validation.** Extension subagent definitions reference profiles. If a referenced profile doesn't exist, the subagent should still load (the profile mismatch is a runtime concern for `task()`, not a load-time failure). Do NOT call `validateProfileReference()` during extension loading — just store the definition.
5. **SubagentOrchestrator transparency.** The orchestrator already calls `subagentManager.loadSubagent(name)`. After this change, that call transparently resolves extension subagents too. No orchestrator changes should be needed — verify with existing tests.
6. **Backward compatibility.** Existing JSON files on disk lack `source`/`sourceExtension` fields. Ensure `loadSubagent()` handles this gracefully (they default to user-owned behavior).
7. **Scope creep.** Do NOT touch `settingsSchema.ts` or add settings-schema support for subagents — that is batch B48.
