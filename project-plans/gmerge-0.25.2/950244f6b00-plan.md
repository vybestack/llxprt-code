# Playbook: Reimplement History Rendering Perf/OOM Mitigation in DefaultAppLayout

**Upstream SHA:** `950244f6b00`
**Upstream Subject:** Attempt to resolve OOM w/ useMemo on history items (#16424)
**Upstream Stats:** UI/performance change; small-to-moderate LLxprt adaptation

## What Upstream Does

Upstream reduces runaway re-render/allocation pressure in the main history UI by memoizing rendered history items instead of rebuilding the full item list on every app render. The intent is to stabilize object identity for expensive history-entry elements and lower memory churn in long sessions, especially while streaming.

## Why REIMPLEMENT in LLxprt

1. `CHERRIES.md` marks this commit **REIMPLEMENT** rather than PICK because the LLxprt perf target is `packages/cli/src/ui/layouts/DefaultAppLayout.tsx`, not upstream `MainContent.tsx`.
2. LLxprt currently maps `history` into `HistoryItemDisplay` elements twice inside `DefaultAppLayout.tsx`: once for alternate-buffer `ScrollableList` data and once for the non-alternate-buffer `Static` list. Both paths eagerly rebuild large arrays of React elements on every render.
3. `DefaultAppLayout.tsx` also recreates `headerElement`, `pendingElement`, and the `ScrollableMainContentItem[]` payload inline, which compounds churn during streaming-state updates, timer ticks, loading-phrase changes, shell focus changes, and dialog visibility changes.
4. `HistoryItemDisplay` is already the dominant rendering unit in LLxprt history. The best adaptation is to memoize the layout’s history-derived element arrays without changing LLxprt’s item model, virtualization contracts, or naming.
5. The user explicitly called out that this batch must be grounded in LLxprt’s current repo and that the history perf target is `DefaultAppLayout.tsx`, preserving LLxprt architecture rather than copying upstream component structure.

## Implementation Steps

1. Read `DefaultAppLayout.tsx` fully and identify every place where `history` and `pendingHistoryItems` are transformed into JSX or `ScrollableMainContentItem[]` data.
2. Separate history-derived rendering work from transient UI state such as `currentLoadingPhrase`, `elapsedTime`, loading indicator state, footer metrics, and dialog visibility.
3. Add `React.useMemo()` for the alternate-buffer history items array.
   - Dependencies should include only inputs that materially affect rendered history rows.
   - Avoid capturing unrelated layout state that would defeat memoization.
4. Add `React.useMemo()` for the non-alternate-buffer `Static` items list.
   - Include the header element only if its dependencies genuinely differ from the history row dependencies; otherwise consider memoizing the header separately.
5. Evaluate whether `pendingElement` should remain eager or become memoized independently.
   - It should remain responsive to `pendingHistoryItems`, `constrainHeight`, focus, shell PTY state, and width changes.
   - Do not accidentally freeze live pending output.
6. Preserve the existing branch structure (`useAlternateBuffer` vs non-alternate-buffer) and current virtualization/list contracts.
7. Run targeted tests if they exist and update them only for observable behavior.
8. Run verification.

## Execution Notes / Risks

- **Key repo fact:** the LLxprt perf target is `packages/cli/src/ui/layouts/DefaultAppLayout.tsx`, not upstream `MainContent.tsx`.
- **Risk:** over-broad dependency arrays will nullify the perf benefit. Keep memo inputs limited to the props/state actually used to render history rows.
- **Risk:** under-broad dependency arrays can produce stale UI for shell focus, terminal width, todo-panel visibility, or slash-command metadata. Re-read every prop passed into `HistoryItemDisplay` before locking dependencies.
- **Risk:** the alternate-buffer path wraps data for `ScrollableList`; preserving stable `key` values is important for virtualization correctness.
- **Do not** redesign `HistoryItemDisplay`, `ScrollableList`, or `UIStateContext` for this batch.
- **Do not** chase upstream component names or create a new `MainContent` abstraction just to mirror upstream.
- **Success criteria:** the history rendering paths keep LLxprt behavior unchanged while reducing needless history-array/element reconstruction during unrelated state updates.
