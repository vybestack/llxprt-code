**Upstream SHA:** `0e955da1710`

**Upstream Stats:** debug/chat command feature; moderate LLxprt adaptation

## What Upstream Does

Upstream adds a `/chat debug` command that exposes internal chat/request debugging information. In upstream, this is tied to nightly-only wiring and request-export plumbing around its current chat stack. The command is intended to help developers inspect what the chat system is doing without manually digging through logs.

At a high level, the upstream feature:
- adds a new `/chat debug` subcommand;
- gathers chat/request diagnostic information from the active conversation stack;
- surfaces the debug output in a user-visible form suitable for development builds.

## Why REIMPLEMENT in LLxprt

1. `CHERRIES.md` marks this as **REIMPLEMENT** because the feature is useful but the upstream wiring does not map directly to LLxprt.
2. The request gives the key architectural constraint: chat debug should adapt to current LLxprt chat command surfaces rather than upstream nightly-only wiring.
3. LLxprt's current slash command is `packages/cli/src/ui/commands/chatCommand.ts`, and it currently exposes checkpoint/history operations (`list`, `save`, `resume`, `delete`, `rename`, `clear`, `restore`) rather than an upstream nightly-debug path.
4. LLxprt already has separate debug-oriented surfaces in the repo, including `/debug` command support, so this batch should reuse current LLxprt diagnostics patterns instead of inventing a nightly gate.
5. If any non-slash CLI/debug route is useful, it must fit LLxprt's existing command architecture rather than assume upstream internal request exporters.
6. The command should report information that LLxprt can actually obtain from current chat/config/client objects in this repo.

## Execution Notes/Risks

- **Risk:** avoid promising exported raw request payloads unless LLxprt already has a stable accessible representation for them.

- **Do not** introduce a nightly-build check copied from upstream unless you first verify LLxprt uses that concept in this path.
