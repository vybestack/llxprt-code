# Playbook: Decouple Hooks UI Visibility from Hooks Execution

**Upstream SHA:** `4c961df3136`
**Upstream Subject:** feat(core): Decouple enabling hooks UI from subsystem. (#16074)
**Upstream Stats:** 5 files changed, 18 insertions, 6 deletions

## What Upstream Does

Upstream separates the concept of "hooks UI is visible" from "hooks actually execute."

Previously, a single combined flag `getEnableHooks()` gated **both** command visibility and execution. That function returns `true` only when `tools.enableHooks` (experimental gate, default `true`) AND `hooks.enabled` (user toggle, default `false`) are both true.

After this change:
- **Hooks UI visibility** (whether `/hooks` appears as a slash command, and whether the `hooks` yargs CLI subcommand is registered) is gated by only `tools.enableHooks` (the experimental gate). This is exposed as a new `getEnableHooksUI()` function.
- **Hooks execution** continues to be gated by `getEnableHooks()` which requires both flags.

**User-visible behavior:** After this change, the `/hooks` slash command and the `hooks` CLI subcommand may be visible and accessible to the user while the HookSystem itself remains disabled (i.e., hooks do not actually execute). This is the intended decoupling — users can discover and configure hooks UI even when hook execution is turned off.

Critically, upstream does **NOT** change `getHookSystem()` or HookSystem creation. The HookSystem is still only created when `enableHooks=true`. The decoupling is purely about **command registration/visibility**, not about making HookSystem available when execution is disabled.

## Two Distinct Command Layers

This plan touches two independent command layers. They share the name "hooksCommand" but are completely different objects in different modules. Do not conflate them.

### Layer 1: yargs/CLI command registration
- **File:** `packages/cli/src/commands/hooks.ts` — exports a yargs `CommandModule` named `hooksCommand`. This is a CLI subcommand (e.g., `llxprt hooks migrate`).
- **Consumed by:** `packages/cli/src/config/config.ts` in the `parseArguments()` function, where it is registered via `yargsInstance.command(hooksCommand)`.
- **This plan:** Gates its registration on `getEnableHooksUI(settings)`.

### Layer 2: Slash-command visibility (interactive UI)
- **File:** `packages/cli/src/ui/commands/hooksCommand.ts` — exports a `SlashCommand` named `hooksCommand`. This is the `/hooks` interactive slash command.
- **Consumed by:** `packages/cli/src/services/BuiltinCommandLoader.ts` in `registerBuiltinCommands()`, where it is included in the command array.
- **This plan:** Gates its inclusion on `this.config?.getEnableHooksUI()`.

Both layers currently include their respective `hooksCommand` unconditionally. Both must be gated on the same UI-visibility flag, but through their respective mechanisms.

## Why REIMPLEMENT in LLxprt

1. LLxprt's `getEnableHooks()` in `settingsSchema.ts` (line 2273) has the same two-flag composition as upstream's pre-change version. The refactoring to extract `getEnableHooksUI()` is directly applicable.
2. LLxprt's yargs registration in `packages/cli/src/config/config.ts` (line 601) currently registers the yargs `hooksCommand` (from `commands/hooks.ts`) **unconditionally**. It needs the new `getEnableHooksUI(settings)` gate.
3. LLxprt's `BuiltinCommandLoader.ts` (line 179) currently includes the slash `hooksCommand` (from `ui/commands/hooksCommand.ts`) **unconditionally** in the command list. It needs to be gated on `config.getEnableHooksUI()` using the same spread-conditional pattern already used for `isSkillsSupportEnabled()` at line 154.
4. LLxprt's core `Config` class (line 2659) has `getEnableHooks()` but lacks `getEnableHooksUI()`. This getter and its backing field/param must be added as a cross-package contract change (see note below).

## Cross-Package Config API Contract

Adding `getEnableHooksUI()` / `enableHooksUI` to the core `Config` class in `packages/core/src/config/config.ts` is a **cross-package contract change**. The `Config` type from `@vybestack/llxprt-code-core` is consumed by:
- `packages/cli/src/services/BuiltinCommandLoader.ts` (typed as `Config | null`)
- `packages/cli/src/config/config.ts` (constructs the `Config` instance)
- Any test file that creates mock `Config` objects via `as unknown as Config`

All consumers that create partial mock configs must add `getEnableHooksUI` to remain type-compatible. The core `Config` class must export the method. The CLI's `config.ts` must pass the new `enableHooksUI` param when constructing `Config`.

```typescript
/**
 * Determines if hooks UI should be visible (commands, status indicators).
 * Gated only by the experimental tools.enableHooks flag.
 */
export function getEnableHooksUI(settings: Settings): boolean {
  return settings.tools?.enableHooks ?? true;
}

/**
 * Determines if hooks should be enabled for execution.
 * Both tools.enableHooks (experimental gate) and hooks.enabled (user toggle) must be true.
 */
export function getEnableHooks(settings: Settings): boolean {
  return getEnableHooksUI(settings) && (settings.hooks?.enabled ?? false);
}
```

This is a pure refactor of the existing logic — `getEnableHooks()` behavior is unchanged.

## Execution Notes / Risks

- **Risk: Yargs chain breakage.** The yargs `hooksCommand` (from `commands/hooks.ts`) is currently chained unconditionally at line 601 (`.command(hooksCommand);`). Moving it to a conditional block requires breaking the chain properly — terminate the chain with `;` and then conditionally call `yargsInstance.command(hooksCommand)`. Look at how `extensionsCommand` is conditionally registered on lines 603-605 for the exact pattern to follow.
- **Risk: BuiltinCommandLoader config type.** `BuiltinCommandLoader` receives `Config | null` in its constructor. Since `getEnableHooksUI()` is a new method on `Config`, the `Config` type in `@vybestack/llxprt-code-core` must export it. The pattern `this.config?.getEnableHooksUI()` handles the null case (returns `undefined`, which is falsy, so the spread produces `[]`).
- **Risk: Cross-package build order.** The core package must be built before the CLI package since the CLI imports from `@vybestack/llxprt-code-core`. The project's `npm run build` already handles this via workspace dependency ordering, but be aware during targeted builds.
- **Do NOT** change `getHookSystem()` — it must still return `undefined` when `enableHooks=false`. This batch only decouples command visibility, not HookSystem creation.
- **Do NOT** change `hooksCommand.ts` (either layer) — the command behavior when invoked is unchanged by this batch.
- **Do NOT** add a `getHooksConfigured()` function that inspects hook definitions — that's not what upstream does. The UI gate is purely the experimental `tools.enableHooks` flag.
- **Batch 7 dependency note:** This batch (B8) comes after B7 (c64b5ec4a3a, HookSystem wrapper methods). B7 does NOT touch `BuiltinCommandLoader.ts` or `settingsSchema.ts`. If B7 adds any new Config methods in `packages/core/src/config/config.ts`, ensure they don't conflict with the `enableHooksUI` additions here. The changes are additive and should not conflict.
