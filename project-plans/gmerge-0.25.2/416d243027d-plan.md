## What Upstream Does

Upstream adds robust process lifecycle management to the test harness:
1. Configurable per-test timeouts via environment variables
2. Automatic cleanup of zombie child processes on test teardown
3. Graceful process termination (SIGTERM then SIGKILL) with configurable grace period
4. Better error messages when child processes hang or timeout
5. PTY process tracking for interactive runs
6. Exit-state tracking on `InteractiveRun`

## Why REIMPLEMENT in LLxprt

LLxprt's `integration-tests/test-helper.ts` already has `getDefaultTimeout()` (line 23), `poll()` (line 29), `InteractiveRun` (line 180), and `TestRig` (line 268) with different signatures and lifecycle semantics than upstream. The test helper already supports:
- Environment-based timeouts (`CI=60s`, `LLXPRT_SANDBOX=30s`, local=`15s`) via `getDefaultTimeout()` at line 23
- `InteractiveRun` with `type()`, `sendText()`, `sendKeys()`, `expectExit()`, `kill()` (lines 194-265)
- PTY process tracking via `_interactiveRuns` array (line 276)
- A `cleanup()` method on `TestRig` (line 724) that already iterates `_interactiveRuns` and calls `run.kill()`

The remaining gaps versus upstream are:
- `InteractiveRun.kill()` (line 247) is a one-liner (`this.ptyProcess.kill()`) with no graceful SIGTERM→SIGKILL escalation
- `InteractiveRun.expectExit()` (line 251) hardcodes `60000` ms instead of using `getDefaultTimeout()`
- No exit-state tracking (`_exited`, `_exitCode`) on `InteractiveRun` — `cleanup()` cannot tell if a run already exited
- `TestRig.run()` has no outer timeout — if the spawned child hangs, the promise never resolves

Upstream also references `GEMINI_CLI_HOME` test isolation (which was SKIP'd) so those references must be omitted.

### 1. Modify: `integration-tests/test-helper.ts`

#### a) Add exit tracking to `InteractiveRun` (near line 181)
Add private fields:
```typescript
private _exited = false;
private _exitCode: number | undefined;
```
Wire the `onExit` handler in the constructor (after the existing `onData` handler at line 186):
```typescript
ptyProcess.onExit(({ exitCode }) => {
  this._exited = true;
  this._exitCode = exitCode;
});
```
This can coexist with `expectExit()`'s own `onExit` handler since `node-pty` supports multiple listeners.

#### b) Upgrade `kill()` to graceful termination (line 247)
Replace the bare `this.ptyProcess.kill()` with:
```typescript
async kill(gracePeriodMs = 5000): Promise<void> {
  if (this._exited) return;
  try {
    this.ptyProcess.kill('SIGTERM');
    const exited = await poll(() => this._exited, gracePeriodMs, 100);
    if (!exited) {
      this.ptyProcess.kill('SIGKILL');
    }
  } catch {
    // Process may already be dead — ignore
  }
}
```

#### c) Use `getDefaultTimeout()` in `expectExit()` (line 251)
Change `expectExit()` to accept an optional timeout parameter:
```typescript
expectExit(timeout?: number): Promise<number> {
  const effectiveTimeout = timeout ?? getDefaultTimeout();
  return new Promise((resolve, reject) => {
    const timer = setTimeout(
      () =>
        reject(
          new Error(
            `Test timed out: process did not exit within ${effectiveTimeout}ms.`,
          ),
        ),
      effectiveTimeout,
    );
    this.ptyProcess.onExit(({ exitCode }) => {
      clearTimeout(timer);
      resolve(exitCode);
    });
  });
}
```

#### d) Improve `TestRig.cleanup()` (line 724)
Update to check `_exited` before calling `kill()`:
```typescript
async cleanup() {
  for (const run of this._interactiveRuns) {
    if (!run['_exited']) {
      try {
        await run.kill();
      } catch (error) {
        if (env['VERBOSE'] === 'true') {
          console.warn('Failed to kill interactive run during cleanup:', error);
        }
      }
    }
  }
  this._interactiveRuns = [];
  // ... rest of cleanup unchanged (golden regeneration, testDir rm)
}
```

#### e) Add outer timeout to `TestRig.run()` (line 447)
Wrap the existing promise at line 596 with a timeout race:
```typescript
const processPromise = new Promise<string>((resolve, reject) => { ... });  // existing
const timeoutMs = getDefaultTimeout() * 4;  // generous outer bound
const timeoutPromise = new Promise<never>((_resolve, reject) => {
```

## Execution Notes / Risks

- **Risk: Double `onExit` handler.** `InteractiveRun` constructor currently registers `onData`. Adding `onExit` in the constructor means `expectExit()` also registers its own `onExit`. Both can coexist since `node-pty` supports multiple listeners, but the exit-tracking handler must be first so `_exited` is set before `expectExit`'s handler resolves the promise.
- **Risk: `kill()` already async.** The existing signature at line 247 is `async kill()`. Callers in the codebase (e.g., `cleanup()` at line 728) already `await run.kill()`. No signature change needed, but the new body is a real async operation (polls for exit). Verify no call sites use `.kill()` fire-and-forget.
- **Do NOT** add `GEMINI_CLI_HOME` or any upstream test-isolation env vars. LLxprt uses `INTEGRATION_TEST_FILE_DIR` for test directory isolation.
- **Do NOT** replace the TestRig wholesale — extend it in place.
- **Keep `getDefaultTimeout()` as the single source of truth** for all timeout defaults.
- **Preserve existing `cleanup()` behavior:** golden regeneration (`REGENERATE_MODEL_GOLDENS`) and test-directory removal must remain intact. The enhancement is additive (check `_exited` before killing).
