# Playbook: Fix Spewie Getter in hookTranslator.ts

**Upstream SHA:** `660368f2490`
**Upstream Subject:** bug(core): Fix spewie getter in hookTranslator.ts (#16108)
**Upstream Stats:** ~1 file, small fix

## What Upstream Does

Upstream fixes a bug in the hook translator where `GenerateContentResponse.text` is accessed via a direct property access (`.text`), but in the GenAI SDK this is actually a getter that may throw or behave unexpectedly when the response has no candidates or has safety-blocked content. The upstream fix changes the code to safely extract text content from the response candidates rather than relying on the top-level `.text` getter.

The "spewie" name in the commit refers to the verbose/noisy error output that occurred when the getter threw — the fix stops the spew by using safe property extraction.

## Why REIMPLEMENT in LLxprt

1. LLxprt's `packages/core/src/hooks/hookTranslator.ts` still uses direct `.text` access — `sdkResponse.text` at line 270 in the `toHookLLMResponse()` method.
2. LLxprt's translator is structurally different from upstream's — it's a class hierarchy (`HookTranslator` abstract base → `HookTranslatorGenAIv1` implementation) at lines 75-354 with LLxprt-specific type definitions.
3. The fix is conceptually identical: stop using the potentially-throwing `.text` getter on `GenerateContentResponse` and instead derive the text from the candidates array, which is already being processed in the same method.
4. The `fromHookLLMResponse()` method at line 306 sets `text: hookResponse.text` — this should be updated to derive text from the reconstructed candidates rather than passing through a potentially problematic value.

## Files to Modify / Create

### 1. Modify: `packages/core/src/hooks/hookTranslator.ts`

#### Fix 1: `toHookLLMResponse()` (line 268-301)

Change line 270 from:
```typescript
text: sdkResponse.text,
```
to derive text from candidates safely:
```typescript
text: (() => {
  try {
    return sdkResponse.text;
  } catch {
    // .text getter can throw when response has no candidates or is safety-blocked
    return undefined;
  }
})(),
```

Or preferably, derive from the candidates being processed in the same method (lines 271-299):
```typescript
// Derive text from candidates instead of using potentially-throwing .text getter
const candidatesData = (sdkResponse.candidates || []).map((candidate) => {
  const textParts =
    candidate.content?.parts
      ?.filter(hasTextProperty)
      .map((part) => part.text) || [];
  return {
    content: {
      role: 'model' as const,
      parts: textParts,
    },
    finishReason: candidate.finishReason as LLMResponse['candidates'][0]['finishReason'],
    index: candidate.index,
    safetyRatings: candidate.safetyRatings?.map((rating) => ({
      category: String(rating.category || ''),
      probability: String(rating.probability || ''),
    })),
  };
});

// Safely extract text: concatenate all text parts from first candidate
const derivedText = candidatesData.length > 0
  ? candidatesData[0].content.parts.join('')
  : undefined;

return {
  text: derivedText || undefined,
  candidates: candidatesData,
  usageMetadata: sdkResponse.usageMetadata
    ? { ... }
    : undefined,
};
```

The simplest correct fix is the try/catch wrapper around `sdkResponse.text`. The more thorough fix derives text from candidates. Choose based on upstream's approach — if upstream uses try/catch, match that; if upstream derives from candidates, do that.

#### Fix 2: `fromHookLLMResponse()` (line 306-325)

At line 309, `text: hookResponse.text` passes through the hook response's text field. This is fine since `hookResponse.text` is a plain string property (not a getter), but verify the constructed `GenerateContentResponse` at line 322 doesn't cause issues when its `.text` getter is later accessed. The `as GenerateContentResponse` cast at line 322 may hide getter behavior.

If the `GenerateContentResponse` type has a getter, the constructed object won't have it — it's a plain object cast. This is actually correct behavior and doesn't need changes unless type errors surface.

## Execution Notes / Risks

- **Risk: Very low.** This is a ~5-line bug fix in a single file.
- **The GenAI SDK's `GenerateContentResponse.text` getter** can throw `Error('No candidates in response')` or `Error('Response was blocked due to safety')` when accessed on responses without valid candidates. The fix must handle this gracefully.
- **Do NOT** change the `LLMResponse` interface shape — the `text?: string` field must remain optional.
- **Do NOT** change the `LLMRequest` translation methods (`toHookLLMRequest`/`fromHookLLMRequest`) — they don't use a getter and are not affected.
- **Do NOT** modify the abstract `HookTranslator` base class — only the `HookTranslatorGenAIv1` concrete implementation.
- **The `hasTextProperty` type guard** (line 92) is already used throughout and is safe — it checks for string-typed `.text` properties on plain objects.
