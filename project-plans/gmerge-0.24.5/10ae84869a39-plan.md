# Playbook: Console → coreEvents/debugLogger Migration

**Upstream SHA:** `10ae84869a394a6214f26e812539f0285af6e72f`
**Upstream Subject:** Migrate console to coreEvents.emitFeedback or debugLogger
**Upstream Stats:** 66 files, 564 insertions, 425 deletions

## What Upstream Does

This commit is a **mechanical refactoring** that removes direct `console.*` usage throughout the codebase and replaces it with two proper abstractions:
- **`debugLogger`** for developer-facing debug/diagnostic messages
- **`coreEvents.emitFeedback(severity, message)`** for user-facing errors/warnings

It also adds an ESLint rule `'no-console': 'error'` to prevent future violations and documents the logging guidelines in `GEMINI.md`.

## LLxprt Adaptation Strategy

This is a **MECHANICAL PATTERN-BASED REFACTORING** across 66 upstream files. Of these, **47 files exist in LLxprt** and need transformation. The 19 missing files are mostly A2A server components that don't exist in LLxprt.

### Substitution Patterns

Apply these exact substitutions in each affected file (case-sensitive):

1. **Developer debug messages:**
   - `console.log(...)` → `debugLogger.log(...)`
   - `console.warn(...)` → `debugLogger.warn(...)`
   - `console.error(...)` (in test/internal code) → `debugLogger.error(...)`

2. **User-facing errors:**
   - `console.error('message')` → `coreEvents.emitFeedback('error', 'message')`
   - Multi-line error logs may need to concatenate strings

3. **Test mocks:**
   - `vi.spyOn(console, 'error')` → `vi.spyOn(coreEvents, 'emitFeedback')`
   - `console.error = vi.fn()` → Mock `debugLogger.warn` or `coreEvents.emitFeedback` depending on context

4. **Imports:**
   - Add `import { debugLogger } from '@vybestack/llxprt-code-core';` (LLxprt uses DebugLogger singleton from `packages/core/src/debug/DebugLogger.ts`)
   - Add `import { coreEvents } from '@vybestack/llxprt-code-core';` for user feedback
   - In core: `import { coreEvents } from '../utils/events.js';` and `import { DebugLogger } from '../debug/DebugLogger.js';`

5. **Special case (AppContainer.tsx):**
   - Remove `console.clear()` call entirely (line ~845) — alternate buffer clearing is handled elsewhere

## Implementation Steps

1. **Preparation:**
   - Run `rg "console\.(log|warn|error|debug)" --type ts --type tsx` to list all violations
   - Cross-reference with the upstream diff to identify files that exist in LLxprt

2. **Add ESLint Rule:**
   - Edit `eslint.config.js` and add `'no-console': 'error'`
   - Run `npm run lint` to see all violations (there will be hundreds initially)

3. **Migrate Core Files (5 files):**
   - Start with `packages/core/src/hooks/hookRunner.ts` and `config.ts`
   - Pattern: `console.error(...)` → `debugLogger.error(...)`
   - Add import: `import { DebugLogger } from '../debug/DebugLogger.js';`
   - Instantiate: `const debugLogger = DebugLogger.getLogger('llxprt:hooks');`

4. **Migrate CLI Main Files (gemini.tsx, nonInteractiveCli.test.ts):**
   - In `gemini.tsx`, add the `CoreEvent.UserFeedback` listener (lines ~722-741 in upstream)
   - Replace `console.error(...)` with `coreEvents.emitFeedback('error', ...)`
   - Update tests to mock `coreEvents.emitFeedback` instead of `console.error`

5. **Migrate UI Components (30+ files):**
   - For each component test, replace `vi.spyOn(console, 'error')` with `vi.spyOn(debugLogger, 'warn')`
   - For user-facing errors in components (e.g., EditorSettingsDialog.tsx), use `coreEvents.emitFeedback`
   - For debug/test logging, use `debugLogger.debug(...)`

6. **Batch Process Tests:**
   - Use a script or manual find/replace to handle common patterns:
     ```bash

     find packages/cli/src -name "*.test.tsx" -exec sed -i '' \
       's/vi.spyOn(console, "error")/vi.spyOn(coreEvents, "emitFeedback")/g' {} \;
     ```
   - **WARNING:** Review each change manually — some test contexts need `debugLogger` instead

7. **Special Cases:**
   - `AppContainer.tsx`: Remove the `console.clear()` call entirely (line ~845)
   - `FileCommandLoader.ts`: All 4 `console.error` → `coreEvents.emitFeedback('error', ...)`

8. **Verification:**
   - Run `npm run lint` — should have zero `no-console` violations
   - Run `npm run test` — all tests should pass
   - Manually test that user-facing errors show in the UI
   - Check that debug logs go to the Ctrl+O console (not the main UI)
