# Plan Review Feedback

Required corrections:

- Remove contradictory/obsolete plan files. `project-plans/issue1591/plan/` contains multiple competing phase sequences and duplicate phase numbers (`01-skeleton.md`, `01-package-skeleton.md`, `03-policy-engine-extraction.md`, etc.) that conflict with the main `00-overview.md` sequence. This violates sequential phase coordination and makes the plan unsafe to execute.
- Fix package boundary contradictions. Several analysis/pseudocode files still explicitly allow or require `packages/policy` to depend on `@vybestack/llxprt-code-core` or deep-import core utilities. Issue #1591 requires the new policy package must not depend on core, providers, tools, or CLI. The executable plan must consistently enforce zero core dependency.
- Fix `@google/genai` contradiction. The specification and P03/P07 allow `@google/genai` as a policy dependency for `FunctionCall`, while other analysis correctly identifies that this couples policy to provider SDK types and proposes `PolicyFunctionCall`. Choose the safe boundary: define policy-owned structural types and remove `@google/genai` from policy production dependencies unless the issue explicitly allows provider SDK coupling.
- Address the missing `packages/settings` requirement honestly. The spec says policy config should depend on `packages/settings`, but the repository has no `packages/settings` workspace. The plan mentions splitting config but does not clearly state the gap as a blocker/constraint with a safe interim strategy. Add an explicit requirement: do not invent `packages/settings`; keep settings/config orchestration in core or use injected interfaces until a settings package exists.
- Add/repair TDD phase structure. Some phases copy existing tests and source together, or describe tests that pass immediately after copying, which is not TDD-first. Each implementation area needs explicit RED behavioral tests before production changes, then implementation phases that make those tests pass.
- Add missing stub phases for all implementation areas. `dev-docs/PLAN.md` requires stub → TDD → implementation cycles. Current main flow has P03 scaffold stub, but policy extraction, confirmation bus extraction, core integration, and test migration do not consistently have separate stub, TDD, implementation, and verification phases.
- Fix confirmation bus type design. The latest plan only makes `ToolCallsUpdateMessage<T = unknown>` generic but keeps `FunctionCall` from `@google/genai`. It should remove direct scheduler/tools/provider type dependencies entirely with policy-owned structural types and clear core mapping points.
- Clarify `PolicyEngine` public entry point and exports with exact API surface. The spec lists `PolicyEngine`, but conflicting analysis files also claim `createPolicyEngineConfig` and `createPolicyUpdater` move to policy while the main spec says they stay in core. Decide and document one clean public interface; preserve core re-exports for backward compatibility.
- Correct backward compatibility/core migration details. The plan says internal core imports should use `@vybestack/llxprt-code-policy`, while some analysis says core production code must not import policy except index re-exports. Since core must consume the extracted package, update the boundary rule to allow core imports from policy and verify no reverse policy→core dependency/cycle.
- Add explicit cleanup/removal or re-export strategy for old core files. Acceptance requires relevant code lives in `packages/policy`; the plan copies files first but does not consistently specify when old core source files become thin re-export shims or are removed, while preserving public deep imports where needed.
- Fix unsafe failure recovery commands. Several phases include `rm -rf packages/policy` and broad `git checkout -- packages/core/`. That conflicts with repo safety practices and can discard unrelated user work. Replace with narrow, reviewed recovery instructions or require coordinator intervention.
- Fix exact verification coverage. Final verification must include all required commands in one final gate: `npm run test`, `npm run lint`, `npm run typecheck`, `npm run format`, `npm run build`, and `node scripts/start.js --profile-load ollamakimi "write me a haiku and nothing else"`.
- Correct dependency declaration details. `@google/genai` is sometimes dependency, sometimes devDependency, sometimes forbidden. `@vybestack/llxprt-code-telemetry` is required in the spec but omitted in some skeleton variants. The package manifest plan must be single, exact, and aligned with boundary decisions.
- Add concrete validation for rule file loading from the new package. The plan copies TOML policy files but needs explicit behavioral tests that `loadDefaultPolicies()` resolves packaged policy files correctly from source and built `dist` output.
- Ensure all phase files use one plan ID. Current files mix `PLAN-20260608-POLICY` and `PLAN-20260608-ISSUE1591`, which breaks marker verification and phase tracking.
