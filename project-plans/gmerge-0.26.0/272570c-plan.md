#### 2. `packages/cli/src/config/config.ts`

Update the skills support logic to check both settings using OR fallback:
```typescript
// Register skills command if enabled
if (settings.experimental?.skills || (settings.skills?.enabled ?? true)) {
  yargsInstance.command(skillsCommand);
}

// In loadCliConfig or equivalent:
skillsSupport:
  settings.experimental?.skills || (settings.skills?.enabled ?? true),
```

**Key OR-fallback logic** (upstream-exact semantics):
- `experimental.skills || (skills.enabled ?? true)`
- If `experimental.skills` is **truthy**, that result is used (backward compat, short-circuits)
- If `experimental.skills` is **falsy** (false, undefined, or absent), falls through to `skills.enabled ?? true`
- **Critical**: `experimental.skills: false` does NOT force skills disabled — it falls through to `skills.enabled` (default `true`), so skills remain **enabled** by default even when `experimental.skills` is false
- Only `skills.enabled: false` explicitly disables skills in the new model

## Notes

The key behavioral change is:
- **Before**: Skills required `experimental.skills: true` to enable
- **After**: Skills are enabled by default; disable with `skills.enabled: false`
- **Backward compat (exact OR logic)**: `experimental.skills || (skills.enabled ?? true)`
  - `experimental.skills: true` → skills on (short-circuits OR)
  - `experimental.skills: false` → falls through to `skills.enabled ?? true` → skills **on** by default
  - `skills.enabled: false` (with no `experimental.skills`) → skills **off**
