## P09 Frozen-Driver Correction

The early integration-first parity slice `packages/agents/src/api/__tests__/cli-turn-parity.early.spec.ts` (authored in P07 as a frozen RED driver) was internally inconsistent and could NEVER pass, even with a perfect `fromConfig`. Two defects, BOTH masked at P07 because the `fromConfig` stub rejected before reaching the tool drive, were corrected in P09 so the SAME behavioral assertions pass GREEN with no weakening.

### The Two Defects

**Defect #1 (confirmation deadlock):** The helper `buildCliStyleConfig` sets `interactive: true` and injects a priority-4.0 forcing-ASK policy. The `parity-toolcall.jsonl` fixture issues a `read_file` tool call. Path B (the reference `AgenticLoop`) is constructed WITH an approval handler returning `ProceedOnce`, but Path A called `fromConfig({ config })` with NO `onApproval` handler → the tool's `TOOL_CONFIRMATION_REQUEST` was never answered → the scheduler waited indefinitely → infinite hang (reproduced: EP1 timed out at 15003ms).

**Defect #2 (fixture starvation):** EP2 and EP3 drove Path A then Path B over the SAME single `built` config whose FakeProvider fixture has a finite 2-turn script. Path A consumed the script; Path B then ran on the drained config and ended with `doneReason: "error"` (2 events) while Path A was healthy (`doneReason: "stop"`, 11 events) → `toStrictEqual` parity failed.

### Coordinator Probe Evidence (proving the defects were in the DRIVER, not in `fromConfig`)

- `fromConfig({ config, onApproval: () => ToolConfirmationOutcome.ProceedOnce })` over `parity-toolcall.jsonl` → drove the tool turn to exactly ONE terminal `done` in ~180ms (NO hang). `fromConfig` is correct.
- With each path given its OWN `buildCliStyleConfig('parity-toolcall.jsonl')` AND Path A given `onApproval: ProceedOnce`, `projectEvents(pathA)` `toStrictEqual` `projectEvents(pathB)` PASSED (tool=`read_file`, one `done`). Full parity holds.

The established headless-approval pattern across this very suite is a caller-supplied `ProceedOnce` handler (see `core-tools.spec.ts` T11 "onApproval auto-answers", `agent-bootstrap.spec.ts:443`, `disposal.spec.ts:123`). A silent auto-approve DEFAULT inside `fromConfig` would be a security smell and is intentionally NOT introduced.

### Exact Edits (assertion-preserving)

1. **EP1** — `fromConfig({ config })` → `fromConfig({ config, onApproval: () => ToolConfirmationOutcome.ProceedOnce })`. `expect(agent.getConfig()).toBe(config)` and the one-terminal-done assertions preserved verbatim.
