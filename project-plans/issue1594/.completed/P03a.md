## 2. Semantic Verification Checklist (MANDATORY)

### Q1 — Do the types match specification.md §Data Schemas exactly (field names/optionality)?

YES. `ProviderAuthSchema` (apiKey/apiKeyFile/keyName/baseUrl/oauth, all optional),
`AgentAuthSchema` (= ProviderAuth + perProvider + profile), and the AgentConfig field
set match the §Data Schemas illustrative shape and the REQ-002 authoritative field set.
`provider` and `model` are REQUIRED (no `?`) in BOTH the interface
(`config-types.ts:146-147`) and the schema (`config-schema.ts:102-103: z.string()` with
no `.optional()`); every other field is optional in both — consistent with the spec.
Host callbacks (`onApproval`, `onOAuthPrompt`, `editorCallbacks`, `toolSchedulerFactory`)
are kept OUT of the Zod object and re-attached via `AgentConfigWithCallbacks =
z.infer<...> & { ...callbacks }`, exactly as §Data Schemas prescribes
(`type AgentConfig = z.infer<typeof AgentConfigSchema> & { onApproval?, ... }`).
The §Data Schemas note "full typed set fixed in REQ-002 table" is honored — the
implemented set is a superset of the illustrative schema and is consistent with it.

### Q2 — Is the classification honored (every §4.2 / ConfigParameters field typed, assigned to a named sub-surface/app-service, or justified in settings-escape-hatch; nothing CLI-critical hidden in `settings`)?

YES. `CONFIG_FIELD_CLASSIFICATION` enumerates all 54 declarative AgentConfig fields:
46 `typed`, 7 `agent-sub-surface`, 1 `settings-escape-hatch`, 0 `app-service`. Every
CLI-critical ConfigParameters field that the spec's grep loop demands
(folderTrust, embeddingModel, debugMode, continueOnFailedApiCall, allowedTools,
coreTools, toolDiscoveryCommand, toolCallCommand, mcpServerCommand, allowedMcpServers,
blockedMcpServers, mcpEnabled, extensionsEnabled, compressionThreshold, projectHooks,
disabledHooks) is present AND classified `typed` — none are hidden in `settings`.
The only `settings-escape-hatch` entry IS `settings` itself, documented UNSTABLE with a
shadow-key throw rule. The 7 `agent-sub-surface` entries are correctly NON-
ConfigParameters runtime concerns: `auth` (precedence resolution), `recording`
(session sub-surface), `extensions` (durable app-service subpath), `onApproval`,
`onOAuthPrompt`, `editorCallbacks`, `toolSchedulerFactory` (host callbacks/factory —
never ConfigParameters fields). Rationale quality is good: each typed entry names its
exact ConfigParameters target (e.g. `policy -> policyEngineConfig`, `shell ->
shellReplacement`, `toolOutputLimits -> truncateToolOutputThreshold/Lines`,
`compression -> chatCompression`), and sub-surface entries explain WHY they are not
ConfigParameters fields.

Spot-check note (advisory, non-blocking): TWO fields — `tools` (entry 5) and `coreTools`
(entry 43) — both claim "Maps to ConfigParameters.coreTools". This is an adapter-phase
reconciliation concern (two public inputs onto one core field), and the rationales DO
differentiate intent ("explicit tool allowlist" for coreTools). It is a documentation/
adapter detail, not a type-surface gap. Recorded as a minor advisory for P05/adapter.

### Q3 — Does `ProviderAuth` express the verified precedence inputs?

YES. `ProviderAuth` carries `apiKey` (raw/inline key), `apiKeyFile` (keyfile),
`keyName` (named key), `baseUrl`, and `oauth`. `AgentAuth extends ProviderAuth` adds

## 3. Schema-First Integrity Finding (RULES.md)

The Zod schema is `.strict()` at every level (11 `.strict()` calls, including
`AgentConfigSchema` and `ProviderAuthSchema`/`AgentAuthSchema`), so parse REJECTS
unknown / shadowed keys as REQ-002 requires.

CONSISTENCY MODEL: `AgentConfig` (config-types.ts) is a HAND-WRITTEN interface; the
schema derives `ParsedAgentConfig = z.infer<typeof AgentConfigSchema>` and
`AgentConfigWithCallbacks = ParsedAgentConfig & { callbacks }`. A field-by-field diff of
the AgentConfig interface vs the schema object shows: every schema field exists in the
interface, and every interface field exists in the schema EXCEPT the four host
callbacks (`onApproval`, `onOAuthPrompt`, `editorCallbacks`, `toolSchedulerFactory`) —
which are INTENTIONALLY schema-excluded (functions are not JSON-validatable) and
re-attached via `AgentConfigWithCallbacks`. This is exactly the §Data Schemas design.

DRIFT RISK (acknowledged, judged LOW/acceptable for this phase): because `AgentConfig`
is hand-written rather than `z.infer`-derived, the interface and schema CAN diverge in
VALUE types. Observed intentional looseness where the schema is broader than the typed
interface:
- `outputFormat`: interface `ConfigParameters['outputFormat']` (OutputFormat enum) vs
  schema `z.string()`.
- `policy`: `PolicyEngineConfig` vs `z.record(z.unknown())`.
- `extensions`: `readonly GeminiCLIExtension[]` vs `z.array(z.record(z.unknown()))`.
- `skills`: `readonly SkillDefinition[]` vs `z.array(z.record(z.unknown()))`.
- `sandbox`: `SandboxConfig` vs `z.record(z.unknown())`.
- `hooks`/`projectHooks`: `AgentHooks` vs `z.record(z.unknown())`.

Severity judgement: ACCEPTABLE for P03. The schema is a structural GATE (reject unknown
top-level keys / wrong primitive shapes) while the TypeScript interface carries the
precise compile-time contract; the looser inner schemas validate "is an object/array"
without re-encoding complex core types (which would itself be re-invention and a
maintenance hazard). The directions are SAFE: the schema is never NARROWER than the
interface (which would reject valid typed input), only broader. No field is typed in the
interface but absent from the schema (other than the deliberately-excluded callbacks).
This satisfies "rigorously in sync field-by-field" for field NAMES and OPTIONALITY, with
documented, deliberate inner-shape looseness. Recommended (advisory, not blocking):
tighten `outputFormat` to `z.nativeEnum(OutputFormat)` and `approvalMode` is already
`z.nativeEnum` — for parity a future hardening pass could narrow the enum-typed fields.

## 4. Holistic Functionality Assessment

### The public AgentConfig surface (in my own words)

`AgentConfig` is an immutable, declarative description of a desired agent. It requires
exactly two fields — `provider` and `model` — and exposes a broad set of optional typed
fields that map 1:1 onto consumer-relevant `ConfigParameters` (tools/coreTools/
excludeTools, mcpServers + the MCP gating/command knobs, approvalMode, working dir,
session id, include dirs, file filtering, telemetry, proxy, max-turns, compression +
threshold, checkpointing, recording, policy, extensions + extensionsEnabled, ide, hooks/
projectHooks/disabledHooks, memory, stream idle timeout, tool-output limits, output
format, shell, context limit, skills, useWriteTodos, sandbox, folderTrust,
embeddingModel, debugMode, continueOnFailedApiCall, interactive, plus the tool-discovery/
call commands). Auth is its own typed sub-object (`AgentAuth`: raw key/keyfile/key-name/
base-url/oauth + per-provider overrides + profile reference). Four host integration
points are typed function/object callbacks kept off the JSON schema (`onApproval`,
`onOAuthPrompt`, `editorCallbacks`, `toolSchedulerFactory`). A single UNSTABLE `settings`
escape hatch carries long-tail ConfigParameters entries, with a documented shadow-key
throw. A companion `config-classification.ts` table records, for every field, whether it
is `typed` / `agent-sub-surface` / `app-service` / `settings-escape-hatch` plus a
rationale, so #1595 can never silently bury a CLI-critical field in `settings`.
