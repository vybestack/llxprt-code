## DESIGN_DECISIONS

### Blocker 1 — schema rejected plain-object MCP configs (ZodError)

`config-schema.ts` previously validated `mcpServers` with
`z.record(z.instanceof(MCPServerConfig))`. `MCPServerConfig` is a ~20-arg
positional CLASS, so any public caller passing a plain object (the natural public
surface) was rejected at `createAgent` before any control logic ran.

**Resolution (genuine, not a test patch):** replaced `z.instanceof` with a
STRUCTURAL `McpServerConfigSchema` (`z.object({...})`) that mirrors the
MCPServerConfig field shape across all transports (stdio/sse/http/websocket) plus
metadata + OAuth fields. No `.strict()` so forward-compatible server keys pass
through. The inferred type is widened at the `AgentConfigSchema` boundary to
`z.ZodType<MCPServerConfig>` so the PUBLIC `AgentConfig.mcpServers` type still
reads cleanly for consumers. Core consumes the parsed plain object structurally
as `MCPServerConfig` (the runtime never requires an actual class instance for the
discovery path), so no coercion shim is needed. This is a strict improvement to
the public surface, not a weakening — a 20-arg class was never an acceptable
public input. P03 schema specs and P14 adapter behavior (exercised through
boundary.spec.ts + createAgent across the api suite) stay green.

**Root-cause note:** the agents specs execute against the BUILT `dist/` of
`@vybestack/llxprt-code-agents` (and transitively core/mcp/ide-integration), NOT
`src/`. The original ZodError persisted after the src fix purely because the
agents `dist` was stale; rebuilding the dependency chain (core → mcp →
ide-integration → agents) is REQUIRED for src schema/control changes to take
effect in the specs.

### Blocker 2 — fakes not wired; real discovery would surface zero tools

The test helpers (`fakeMcpServer.ts`, `fakeIde.ts`) are standalone in-memory
fakes; nothing connected their served tools / IDE state to the real agent, and
real discovery would spawn a non-existent stdio binary.

**Resolution — shipped env-driven test doubles (FakeProvider analogue):** added
two SHIPPED seams that live in production packages (never in `__tests__`), are
legitimate test doubles exactly like the shipped `FakeProvider`
(`LLXPRT_FAKE_RESPONSES`), and drive the REAL discovery/IDE machinery:

- **MCP:** `packages/mcp/src/fake/fakeMcpDiscovery.ts` — `isFakeMcpDiscoveryActive()`
  (reads `LLXPRT_FAKE_MCP`), `loadFakeMcpFixture()`, `applyFakeServerDiscovery()`,
  and a real `FakeMcpTool extends BaseDeclarativeTool` (carries a non-empty
  `serverName`, inert execute). `McpClientManager` gained a fake discovery branch
  (`connectAndDiscoverFake`) that registers the fixture's tools into the REAL
  `ToolRegistry`, drives the REAL `MCPDiscoveryState` state machine, records REAL
  per-server failures (`discoveryFailures` + `getDiscoveryFailures()`), and a
  `whenDiscoverySettled()` awaiter the gate uses. No process is spawned; the REAL
  registry/status/discovery-state paths run end-to-end.
- **IDE:** `packages/ide-integration/src/ide/fake-ide-environment.ts` —
  `isFakeIdeActive()` (reads `LLXPRT_FAKE_IDE`), `loadFakeIdeFixture()`,
  `trustFakeIde()`. `IdeControl` reads the fixture through the REAL control logic
  when the seam is active, else falls back to `detectIdeFromEnv()`.

The test helpers were extended (allowed infra work) to SERIALIZE their state to
the fixture JSON and set the env var on construction + every mutation; the LOCKED
behavioral assertions in `mcp-discovery.spec.ts` / `ide.spec.ts` are unchanged.
Production code imports nothing from `__tests__` (verified).

### Discovery gate + error surface

`awaitMcpDiscoveryGate(opts)` in `agentImpl.ts`: default AWAITS
`manager.whenDiscoverySettled()` before the model turn; `mcpDiscovery:'skip'` opts
out without awaiting; on recorded failure returns a typed
`AgentError{code:'mcp_discovery_failed', message}`. `stream()` yields exactly ONE
`done{reason:'error'}` and stops (the stream error event payload carries no
`code`). `chat()` runs the gate once, surfaces the typed code through the buffered
`AgentResult.error.code`, then re-streams with `mcpDiscovery:'skip'` to avoid
double-gating. `mcp.status()/discoveryState()` and `listTools()` remain callable
(non-blocking) while discovery is pending.
