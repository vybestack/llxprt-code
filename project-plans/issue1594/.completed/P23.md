## DESIGN_DECISIONS

### REQ-015 — hooks as a real control surface (control/hooks.ts)

Extracted the former inline `SafeHookControl` stub (agentImpl.ts ~1053, where
`onHookExecution` threw NotYetImplemented and the triggers were safe no-ops) into a
new production `HookControl implements AgentHookControl` in
`packages/agents/src/api/control/hooks.ts`, mirroring the other control surfaces
(mcpControl.ts/ideControl.ts) and wired via `buildHookControl()` in AgentImpl with an
injected `HookControlDeps` bundle `{ config, messageBus, sessionId, cwd }`.

Two genuine observation sources feed the public `onHookExecution(cb)` observers:

1. **Lifecycle triggers.** `triggerSessionStart()` / `triggerSessionEnd()` call the
   REAL core triggers `triggerSessionStartHook(config, SessionStartSource.Startup)` /
   `triggerSessionEndHook(config, SessionEndReason.Exit)` (lifecycleHookTriggers.ts),
   which fire `config.getHookSystem()?.fireSessionStartEvent/...EndEvent` when hooks
   are enabled. Each trigger then emits an observable agent-facing
   `{event,input}`/`{event,output}` pair to registered observers so the CLI can
   render the lifecycle transition EVEN when no command hooks are registered (the
   headless fake agent has `getEnableHooks()===false`, so the core trigger returns
   undefined — the emission is what the locked T15b asserts on, via the real event
   ordering, not a canned value). The output is projected from the trigger's
   aggregated result when present, else a well-formed `continue:true/decision:'allow'`
   default.

2. **Bus-mediated executions.** The control subscribes to the SHARED MessageBus
   (`MessageBusType.HOOK_EXECUTION_REQUEST` / `HOOK_EXECUTION_RESPONSE`), buffers each
   request by `correlationId` in `pendingByCorrelation`, and on the correlated
   response forwards the pair to observers. This reuses the single shared MessageBus
   the runtime threads through every surface (B2 invariant) — NO second bus is
   created (the deps bundle receives `this.deps.messageBus`).

`clear()` resets the observer set + the correlation buffer; a fresh
`onHookExecution` registration after `clear()` still returns a working unsubscribe
(the control stays bus-subscribed for the agent's lifetime). `detach()` (called from
dispose) tears down the two bus subscriptions so they do not outlive the agent.

The three HookExecution shapes are bridged cast-free: bus payloads carry only
`{eventName, correlationId}`, so the public input is rebuilt from the documented
HookInput base fields (`session_id` via `deps.sessionId()`, `cwd` via `deps.cwd()`,
`hook_event_name`, ISO `timestamp`) and the event-name string is narrowed to
`HookEventName` by membership scan (defaulting to `Notification`). No `as`/`as any`.

`createAgent.ts` already called `await agent.hooks.triggerSessionStart()` after the
facade build (the P15 placeholder); that call is now real and observable. It does not
throw for hooks-less agents because the core trigger short-circuits on
`getEnableHooks()===false` and the emission path is guard-free.

### REQ-006 + REQ-016 — CONDITIONAL T19 injected-factory scheduler teardown

Per dispose.md lines 40-47, ONLY a facade-held scheduler created via the injected
`toolSchedulerFactory` is disposed by `dispose()`; per-turn loop schedulers remain
owned+disposed by AgenticLoop (`config.getOrCreateScheduler` → `config.disposeScheduler`
in the loop's finally) and dispose() must NOT touch them.

Implementation: `wrapSchedulerFactory` (agentBootstrap.ts) — already wrapping the
injected `AgentSchedulerFactory` into a core `ToolSchedulerFactory` (P17) — now also
RECORDS each created `AgentSchedulerHandle` into a shared `injectedSchedulerHandles`
array. That array is threaded `createAgent → recordOwnership → OwnershipRecord
.injectedSchedulerHandles`. `AgentImpl.dispose()` iterates the array AFTER aborting
the active run and disposes each handle (best-effort, awaited). The caller-owned
factory FUNCTION is never disposed (it has no dispose; only its produced handle
instances are). When no factory is injected the array is empty and the loop is a
no-op (most agents have no facade-held scheduler).

**Deliberate boundary with P24:** handles are stored on the OwnershipRecord array,
NOT exposed as the impl fields (`injectedFactoryScheduler` / `confirmationCoordinator`)
that disposalProbe.ts reads. Wiring those probe-readable fields + AggregateDisposeError
+ the LSP/extensions/session-lock completion markers is P24's job. The T19 slice
therefore does NOT satisfy disposal.spec.ts — verified RED for the right reason
(clean `expected false to be true` assertion failures, not crashes).

### REQ-002 — sandbox status design decision (NO untested public API added)

The P23 REQ text and the 23a checklist item 5 mention exposing sandbox STATUS
(active/command/image). I investigated every spec for a sandbox-status accessor:
`grep -rn "sandbox" packages/agents/src/api/__tests__` finds ONLY sandbox-boundary.spec.ts
(T18e), which asserts sandbox is accepted at createAgent and the agent runs a turn —
there is NO RED test asserting a `agent.sandbox.status()` / sandbox-status accessor.

**Decision:** do NOT add an untested public API method. Adding a public accessor with
no governing RED test would violate the plan's own TDD discipline (every public
surface must be driven by a locked behavioral test). Sandbox remains accepted-at-
construction instance config (`AgentConfig.sandbox: SandboxConfig`, config-types.ts /
schema) that routes execution; live mutation is intentionally NOT a runtime method
(enforced at compile time — the recreate path is app-service/P27). T18e stays GREEN
unchanged. This REQ-text-vs-absent-test gap is recorded here for the verifier: if a
future phase adds a sandbox-status RED spec, the accessor should be added then.
