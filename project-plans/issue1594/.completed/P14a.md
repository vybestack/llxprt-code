## 2. Semantic verification checklist

**Q1 — Is `toConfigParameters` genuinely pure (input not mutated/referenced, output frozen)?**
YES. Input is only read. Output is a brand-new `params` object. Arrays are copied
with spread (`[...config.tools]` etc.), records/objects deep-cloned via
`clone()` (JSON round-trip), so no input reference leaks into the output.
Scalars/strings are immutable. Result is `Object.freeze(params)`. Evidence:
`agentConfig.adapter.ts` `applyArrayMappings` (spread), `applyObjectMappings`/
`clone()` (deep clone), `toConfigParameters` final `Object.freeze`.

**Q2 — Does the settings shadow-check throw `AdapterError` on collision with a typed field?**
YES. `applySettings` iterates `Object.entries(config.settings)`; if
`TYPED_FIELD_SET.has(k)` it throws `new AdapterError('field ' + k + ' must be a
typed AgentConfig field, not settings')`. `TYPED_FIELD_SET` is built from every
classification entry with `classification === 'typed'`. Evidence: lines for
`TYPED_FIELD_SET` and `applySettings`.

**Q3 — Does `mapLoopStream` guarantee EXACTLY ONE `done` on all terminal paths and ZERO for standalone non-terminal Blocked?**
YES. State machine traced:
- Inline-terminal variants (UserCancelled, AgentExecutionStopped, Finished) yield `makeDone` and set `emittedDone=true`; the loop-end block is then skipped (`!emittedDone` false).
- Reason-pending terminal variants (Error, StreamIdleTimeout, ContextWindowWillOverflow, LoopDetected, MaxSessionTurns) set `pendingDoneReason` and do NOT emit done inline; loop-end block emits exactly one `makeDone(state, pendingDoneReason ?? 'stop')`.
- Standalone AgentExecutionBlocked: `isStandaloneBlocked` true → `sawActivity` stays false; it does not set `pendingDoneReason`; loop-end guard `(sawActivity || pendingDoneReason!==null)` is false → ZERO done. Confirmed by RED test "AgentExecutionBlocked → hook-blocked (NON-terminal)" asserting `doneEvents.length === 0`.
- Blocked-then-Content-then-Finished: Content sets `sawActivity=true`, Finished emits the single done; RED test confirms exactly one done with correct ordering.

**Q4 — Are all 21 variants mapped to public projections with no internal-type leakage; Stopped/Blocked read FLAT?**
YES. `projectToolCall` returns `{id,name,args}` (AgentToolCall) — drops
`prompt_id`/`isClientInitiated`. `projectToolResult` returns
`{id,name,output,isError}` (AgentToolResult) — no raw ToolCallResponseInfo. The
public `AgentEvent` union (event-types.ts) carries only sanitized public types.
`toStopInfo(e)` reads `e.reason / e.systemMessage? / e.contextCleared?` directly
off the FLAT event, never `e.value`. Confirmed against turn.ts lines 274-285
where both Stopped/Blocked are flat `{type, reason, systemMessage?, contextCleared?}`.

**Q5 — Do the P10 characterization tests pass for the pure-mapping variants?**
YES. Full suite GREEN: 38/38 (see command output). Includes the real-loop
abort/awaiting_approval rows and all 21 pure-mapping rows.

## 5. Holistic functionality assessment

**Data flow (AgenticLoopEvent → AgentEvent):** `mapLoopStream` consumes the
internal `AgenticLoopEvent` async stream. Each event is dispatched by `kind`:
`stream`-wrapped events delegate to `mapStreamEvent` (the 21-variant
GeminiEventType table), while loop-native kinds (`tool_update`, `tool_output`,
`tools_complete`, `awaiting_approval`) map directly to `tool-status`/
`tool-result`/`tool-confirmation`. The adapter maintains a small state machine
(`sawActivity`, `emittedDone`, `pendingDoneReason`, `lastFinished`, `lastStop`).
Terminal variants either emit `done` inline (UserCancelled/Stopped/Finished) or
stash a `pendingDoneReason`; at loop end exactly one terminal `done` is
synthesized unless the only thing seen was a non-terminal hook-block. Internal
scheduler types are projected to sanitized public shapes throughout.

**Suspect variant mappings:** None. The only judgement call —
`mapFinishReason` collapsing to `'stop'` — is consistent with the public
`DoneReason` union and the RED contract, so it is correct, not suspect. The two
FLAT events are correctly handled (the historically dangerous spot), and the
exactly-one-done invariant is provably enforced by the state machine.

**Purity & fidelity:** Both adapters are genuinely pure. `toConfigParameters`
never mutates its input, clones all reference-typed fields, and freezes its
output. `mapLoopStream`/`mapStreamEvent` are pure projectors over their input
stream with no external side effects (state is per-call-local). Both are
faithful to their pseudocode specs.
