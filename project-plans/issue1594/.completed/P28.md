## Post-Completion Accuracy Remediation (two corrections)

An independent verification of P28 found two accuracy defects where the doc (or
the map it cites as source-of-truth) described behavior/targets that did not
match shipped source. Both were corrected; the doc now matches reality.

### Correction A — Confirmation section was factually wrong

The section formerly titled "Tool Confirmations and No-Handler Safe Denial"
claimed that when **no** handler is wired the public path performs a safe
`Cancel` denial. That is FALSE per source:

- No `onApproval` → `createAgent` leaves `approvalHandler` undefined
  (`packages/agents/src/api/createAgent.ts:258-259`), so
  `AgenticLoop.wireApprovalHandler()` returns a **no-op** and does NOT
  auto-`Cancel` (`packages/agents/src/core/agenticLoop/AgenticLoop.ts:207-210`).
- The safe `Cancel` denial happens ONLY when a **wired** handler rejects/throws
  (`AgenticLoop.ts:232-248`).
- With no handler in non-interactive mode the confirmation coordinator THROWS
  "requires user confirmation, which is not supported in non-interactive mode"
  (`packages/agents/src/scheduler/confirmation-coordinator.ts:319-325`),
  surfacing an AgentErrorEvent + one `done{reason:'error'}`.

Actions taken:
- Renamed heading to "Tool Confirmations and Safe Denial".
- Rewrote the body to state the REAL behavior: wired-handler-reject → safe
  denial; no-handler → clear error (not silent success, not silent Cancel),
  with a note that an automatic no-handler safe-denial is a possible future
  enhancement not shipped today.
- Rewrote the B7 "Recorded Decisions" entry to match.
- `@plan:PLAN-20260617-COREAPI.P28` + `@requirement:REQ-020` intact; the
  `grep -qiE "safe denial|safely den"` still matches via the handler-reject path.

### Correction B — Dangling production map target `agent.session.save`

`packages/agents/src/app-services/command-api-map.ts` mapped `/chat save` →
`target: 'agent.session.save'`, but `AgentSessionControl`
(`packages/agents/src/api/agent.ts:286-296`) has **no** `save()` — it exposes
`createCheckpoint(label?)`, `restoreCheckpoint`, `listCheckpoints`, `resume`,
`setRecording`, `getRecording`. The only nearby `save()` is on
`AgentAuthKeysControl` (agent.ts:241), unrelated.

Action taken:
- Changed the `/chat save` target to `agent.session.createCheckpoint` (the real
  checkpoint/save method, agent.ts:291); kept `kind:'runtime'` and the existing
  note; preserved the file's `@plan`/`@requirement` markers.
