## Runtime-free seams used

### Providers — `ProviderManager.listProviders()` via the composition factory
- **Seam:** `createProviderManager(context, {})` from
  `@vybestack/llxprt-code-providers/composition.js`
  (`packages/providers/src/composition/providerManagerInstance.ts:506`,
  re-exported at `packages/providers/src/composition/index.ts:20`).
- **Context:** a fresh `SettingsService` (no-arg constructor,
  `packages/settings/src/settings/SettingsService.ts:46`) wrapped in a minimal
  `ProviderRuntimeContext`
  (`packages/core/src/runtime/providerRuntimeContext.ts:46`).
- `createProviderManager` registers all built-in/alias providers and sets a
  default active provider (`gemini`) internally — the caller does NOT need to
  activate anything, and no network/auth turn is triggered. We then read the
  registered NAMES via `manager.listProviders()`
  (`packages/providers/src/ProviderManager.ts:1180`) — the SAME accessor the
  instance path re-projects.
- This is the exact pattern already proven for issue #1594 in
  `createAgent.ts` (`registerProvidersOntoManager`, ~`createAgent.ts:448-456`),
  which builds a manager from a fresh-SettingsService context and enumerates
  `registered.listProviders()`.
- **Projection:** inlined a pure `projectProviderInfos(names)` (name +
  `configured`) rather than importing `buildProviderInfos` from
  `./agentBootstrap.js`. agentBootstrap imports `../core/client.js`
  (`AgentClient`) and the full bootstrap runtime chain; importing it into this
  pre-agent module would pull that heavy chain in for a trivial map. The spec
  explicitly permits inlining an equivalent pure projection to avoid that
  coupling.
- **`configured` semantics:** `false` for every provider. Static, runtime-free
  discovery has no bound credentials and no active runtime, so no provider is
  authoritatively "configured" before an Agent exists. This is an honest
  pre-agent projection (enumeration is the point), not a fabricated per-provider
  literal.

## Spec-vs-reality resolution (why NOT the global runtime accessor)

The phase spec's literal hint — "project from
`@vybestack/llxprt-code-providers/runtime.js` `listProviders`" — is a TRAP for
the static path. That runtime accessor
(`packages/providers/src/runtime/runtimeAccessors.ts:535`) delegates to
`getCliRuntimeServices().providerManager.listProviders()`, and
`getCliRuntimeServices()` (same file, ~142) THROWS when no Config/runtime is
registered. Static discovery runs BEFORE any agent/runtime exists, so wiring the
global accessor would make `listProviders()` throw in its primary use case.

Therefore the static path deliberately reaches the SAME
`ProviderManager.listProviders()` accessor through the runtime-free composition
seam (`createProviderManager` + fresh `SettingsService`) instead of the throwing
global runtime accessor. No call to `getCliRuntimeServices` /
`@vybestack/llxprt-code-providers/runtime.js` exists in `discovery.ts` (the only
mention is an explanatory comment documenting this decision).
