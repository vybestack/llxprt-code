# Plan Review Round 8: Issue #1592 Extract packages/agents

## Verdict: REVISE

The plan is substantially stronger than a typical extraction plan: it is integration-first, identifies real consumers, explicitly forbids shims, and includes dependency-direction and behavior-preservation gates. I verified many of its claims against the working tree rather than relying on the analysis documents.

However, I found several issues that should be corrected before execution. The most important are: (1) the package scaffold phase deliberately does **not** run the authoritative full battery despite the overview saying every code-changing phase must; (2) the verification regexes for workspace import inventories only handle single-quoted forms in P03/P03a even though P02 correctly requires both quote styles; (3) `TaskToolRegistration` naming is inconsistent in P01; and (4) the plan still contains a factually wrong rationale that `Config`'s constructor constructs an `AgentClient`. None of these invalidates the extraction architecture, but they are execution-risky and should be fixed in the plan before handing it to implementation workers.

## Findings

### MAJOR: P02 violates the plan's own authoritative full-battery rule for every code-changing phase

**Problem:** `plan/00-overview.md` defines an authoritative verification battery and says every code-changing phase, including P02, ends with the full battery and every verification phase re-runs it (`project-plans/issue1592/plan/00-overview.md:58-60`). The battery includes `npm run format`, `git diff --exit-code`, `npm run typecheck`, `npm run build`, `npm run test`, `npm run lint`, and the smoke test (`project-plans/issue1592/plan/00-overview.md:62-70`). But P02's verification commands only run `npm install`, `npm run build`, `npm run typecheck`, `npm run lint`, a grep, and `npm run test:scripts` (`project-plans/issue1592/plan/02-package-scaffold.md:48-58`). P02 omits `npm run format`, `git diff --exit-code`, the full workspace `npm run test`, and the smoke test.

**Why this matters:** P02 modifies package metadata, workspace ordering, CI/release files, Dockerfile, tests, and lockfile inputs (`project-plans/issue1592/plan/02-package-scaffold.md:33-45`). Those changes can break the full workspace test runner, formatting, or startup before any code is moved. The overview explicitly says phase docs may add scans but never remove battery items (`project-plans/issue1592/plan/00-overview.md:58-60`).

**Recommendation:** Update P02 and P02a to run the full battery exactly as defined in 00-overview, then add `npm install`, `npm run test:scripts`, and release wiring greps as additional checks.

### MAJOR: P03/P03a dependency inventory regexes miss double-quoted imports even though the plan says all import forms must be covered

**Problem:** P02 correctly instructs workers to generate dependency inventory covering static imports, export-from, dynamic imports, require, and `vi.mock`, with single **and** double quotes (`project-plans/issue1592/plan/02-package-scaffold.md:23-28`). But P03's post-rewrite inventory command only matches single-quoted static `from` imports (`project-plans/issue1592/plan/03-code-move.md:50`), and P03a's workspace-leakage gate only matches single-quoted forms (`project-plans/issue1592/plan/03a-code-move-verification.md:20-24`).

**Evidence from the codebase:** The plan itself asks to scan package.json dependency sections and tsconfig paths (`project-plans/issue1592/plan/03a-code-move-verification.md:24`), which are double-quoted JSON strings; P03/P03a's single-quote-only commands will not see them. The codebase also already uses mixed import/mock forms; for example, a2a imports core package symbols in source (`packages/a2a-server/src/config/config.ts:12-25`), and package metadata/workflow/script wiring is JSON/YAML/JS with quoted package specifiers.

**Why this matters:** This extraction's dependency boundary is one of the core acceptance criteria: core must not depend on agents; agents must not depend on providers or CLI. A generated import inventory that misses quote styles or import forms can incorrectly approve a package.json dependency, a tsconfig path alias, or a moved test mock path.

**Recommendation:** Use the P02 regex consistently in P03 and P03a, or better, use a small script/AST parser that extracts package specifiers from `.ts`, `.tsx`, `.js`, `.json`, vitest configs, and tsconfig files. Ensure the hard-fail checks cover static imports, `export ... from`, dynamic `import()`, `require()`, `vi.mock()`, JSON dependency sections, tsconfig `paths`, and vitest aliases.

### MAJOR: `TaskToolRegistration` is sometimes called `taskToolFactory`, which can cause the wrong seam to be implemented

**Problem:** The integration contract defines a descriptor-style `TaskToolRegistration` because a bare factory would lose `ToolRecord.toolClass`, class-name/static-name matching, and constructor-args semantics (`project-plans/issue1592/analysis/integration-contract.md:41-76`). P01's requirements correctly describe the descriptor approach (`project-plans/issue1592/plan/01-contracts-inversion.md:26-39`). But P01 implementation task 3 says to add `agentClientFactory`, `toolSchedulerFactory`, and `taskToolFactory` to ConfigParameters (`project-plans/issue1592/plan/01-contracts-inversion.md:53-54`). That name contradicts the contract and suggests a bare factory rather than the descriptor.

**Evidence from actual code:** `toolRegistryFactory.ts` currently relies on concrete class metadata: `buildRegisterCoreTool` reads `ToolClass.name` and `ToolClass.Name` (`packages/core/src/config/toolRegistryFactory.ts:101-105`), stores `toolClass`, `toolName`, `displayName`, and `args` in `ToolRecord` (`packages/core/src/config/toolRegistryFactory.ts:131-138`), and uses both class and static names in allow-list/exclude behavior (`packages/core/src/config/toolRegistryFactory.ts:112-124`). The current TaskTool path passes the class itself into `registerCoreTool` (`packages/core/src/config/toolRegistryFactory.ts:247-250`) and the disabled/missing-manager record also stores `toolClass: TaskTool`, `toolName: 'TaskTool'`, `displayName: TaskTool.Name || 'TaskTool'`, and args (`packages/core/src/config/toolRegistryFactory.ts:251-259`). The integration contract's descriptor is justified by this code.

**Why this matters:** A worker following the `taskToolFactory` wording could implement a simple instance factory and silently break `allPotentialTools` metadata, `coreTools` allow-list matching, settings UI metadata, or disabled-tool records.

**Recommendation:** Rename the ConfigParameters field consistently to `taskToolRegistration` everywhere in the plan. Explicitly forbid a bare `taskToolFactory` field. Add a P01/P01a grep or type check that `TaskToolRegistration` is the field type and that `ToolRecord.toolName === 'TaskTool'` and `displayName === 'task'` in the registered path.

### MAJOR: P01 contains a factually wrong rationale about `Config` construction that could mislead implementation

**Problem:** P01 says the constructor param is required for `agentClientFactory` "since the constructor constructs a client today" (`project-plans/issue1592/plan/01-contracts-inversion.md:53-54`). That is not true. The actual `Config` constructor only calls `applyConfigParams` (`packages/core/src/config/config.ts:103-107`), and `applyConfigParams` applies settings/identity/tool governance/runtime/lifecycle fields (`packages/core/src/config/configConstructor.ts:466-476`). `AgentClient` construction happens later in `initialize()` (`packages/core/src/config/config.ts:196-198`) and in `initializeContentGeneratorConfig()` (`packages/core/src/config/config.ts:306-315`).

**Why this matters:** The overall design is correct that factories can be provided as constructor parameters and absence should be a use-time error. The problem is the rationale: an implementation worker may add construction-time validation or otherwise alter non-initializing `new Config(...)` tests because they believe the constructor already crosses the seam. That would violate the integration contract, which correctly says factory absence is a use-time error and non-initializing Config sites must keep working.

**Recommendation:** Replace the incorrect sentence with: "constructor params are required because composition roots already pass `ConfigParameters`, and both later use sites (`initialize()` and `initializeContentGeneratorConfig()`) need the factory before they construct an AgentClient; absence is checked only at those use sites."

### MAJOR: P02 scaffolds an empty package and release/publish wiring before any real code, which is acceptable only if the full battery includes pack/publish sanity and no consumers import it until P03

**Problem:** The plan intentionally scaffolds `packages/agents/src/index.ts` with only `export {};` and forbids any import from `@vybestack/llxprt-code-agents` until P03 (`project-plans/issue1592/plan/02-package-scaffold.md:30-34`). It also adds release/sandbox/Docker/version/test wiring in P02 (`project-plans/issue1592/plan/02-package-scaffold.md:36-45`). This can end green if no consumer imports agents yet, so it is not an isolated-feature failure by itself. But it means release/package wiring for an empty package is temporarily present, and P02's verification does not include the full battery (see first finding).

**Evidence from codebase/provider precedent:** The root workspace currently lists auth, settings, telemetry, mcp, core, providers, cli, a2a-server, test-utils, vscode, and lsp; no agents yet (`package.json:8-19`). Provider release/sandbox touchpoints exist in `scripts/build_sandbox.js` for package dir, pack command, and tarball copy/cleanup (`scripts/build_sandbox.js:97`, `scripts/build_sandbox.js:159-165`, `scripts/build_sandbox.js:225-227`), in `scripts/version.js` (`scripts/version.js:50`), and in release process tests (`scripts/tests/release-process.test.js:70`, `scripts/tests/release-process.test.js:101-129`, `scripts/tests/release-process.test.js:171`, `scripts/tests/release-process.test.js:191`, `scripts/tests/release-process.test.js:221`, `scripts/tests/release-process.test.js:247`). The plan correctly identifies these touchpoints (`project-plans/issue1592/plan/02-package-scaffold.md:36-45`).

**Recommendation:** Keep the scaffold-before-move strategy, but require P02 to run the full battery plus `npm pack --dry-run -w @vybestack/llxprt-code-agents`, `npm run test:scripts`, and all release-process tests. Add a statement that P02 is intentionally not functionally complete but not considered complete until release/test/bundle wiring is verified.

### MINOR: Dependency-direction rules are mostly sound, but the plan should explicitly reconcile move-set inventory vs broad `core/` directory scans

**Problem:** The spec and move-map explicitly keep several files in `packages/core/src/core` (`contentGenerator.ts`, `prompts.ts`, `tokenLimits.ts`, `logger.ts`, `geminiRequest.ts`) (`project-plans/issue1592/specification.md:30-42`, `project-plans/issue1592/analysis/move-map.md:17-20`). A broad inventory over all of `packages/core/src/core` can therefore produce false dependency/package requirements. For example, `logger.ts` stays in core (`project-plans/issue1592/specification.md:40`) but imports settings types in source/tests (`packages/core/src/core/logger.ts:12` per grep), so including it in the agents move-set inventory would incorrectly suggest agents needs that source file/dependency.

**Why this matters:** P02/P03 dependency inventory must be based on the actual move-set, not every file under `packages/core/src/core`, because documented stayers live in that directory. The plan says use the move-set files, but P00a's inventory starts with `ls packages/core/src/core packages/core/src/agents packages/core/src/scheduler` (`project-plans/issue1592/plan/00a-preflight-verification.md:9-16`), so workers need an explicit warning not to feed all listed core files into package dependency generation.

**Recommendation:** Add an explicit note in P00a/P02: dependency inventory must exclude documented STAYS files in `packages/core/src/core`, not simply scan all of `packages/core/src/core`.

### MINOR: `geminiRequest.ts` claim should say "only API consumer," not "only consumers"

**Problem:** The move-map says `geminiRequest.ts` has only `tools/glob.test.ts:9` and `index.ts:81` as consumers (`project-plans/issue1592/analysis/move-map.md:17`). I verified the functional API consumer claim, but grep also finds guard tests mentioning the file/name (`packages/core/src/core/__tests__/config-regression-guard.test.ts:83`, `packages/core/src/core/__tests__/providerAgnosticNaming.test.ts:47`). They do not import the API, so the stay decision remains correct, but the wording is slightly overbroad.

**Recommendation:** Rephrase to "only API/import consumers are...; guard tests also mention the file/name and must be updated only if their assertions require it."

## Integration-first assessment

The plan is integration-first. It lists existing consumers and composition roots in the specification (`project-plans/issue1592/specification.md:102-115`), identifies old code to remove (`project-plans/issue1592/specification.md:116-120`), and defines user access points that must remain unchanged (`project-plans/issue1592/specification.md:122-124`). It also makes P03 atomic: move implementations, remove core exports, add consumer dependencies, flip CLI/a2a imports, and wire factories in one change set (`project-plans/issue1592/plan/03-code-move.md:23-51`). This is not an isolated package build in the PLAN.md sense because P03 cannot end green without updating real consumers.

The one caveat is P02's empty package scaffold. It can build in isolation by design (`project-plans/issue1592/plan/02-package-scaffold.md:30-34`), but it is a preparatory packaging phase, not the implementation of the feature. That is acceptable if the plan keeps the strict no-import-until-P03 rule and fixes P02's verification gap.

## Dependency-direction assessment

The target direction is sound: agents may depend on core and allowed lower packages; core must not depend on agents; agents must not depend on providers or CLI (`project-plans/issue1592/specification.md:13-25`, `project-plans/issue1592/plan/00-overview.md:51-56`). The plan correctly recognizes current provider couplings that force deviations: provider production imports `core/prompts.js` and tests import `ChatSession`, so prompts stay and the ChatSession-dependent provider test must move/rewrite. It also correctly prohibits core tests from importing agents (`project-plans/issue1592/analysis/integration-contract.md:77-81`) and requires dependency scans to include tests (`project-plans/issue1592/plan/00-overview.md:72-78`).

The main risk is not architecture but tooling: P03/P03a import inventory scans are weaker than P02's stated requirements and should be fixed before execution.

## Phase ordering / construction inversion feasibility

The construction-inversion design is feasible and behavior-preserving if implemented exactly:

- `Config` can store factory fields from `ConfigParameters` because constructor parameter application already centralizes assignment (`packages/core/src/config/config.ts:103-107`, `packages/core/src/config/configConstructor.ts:466-476`).
- Factory absence can be a use-time error because `AgentClient` is not constructed until `initialize()` and `initializeContentGeneratorConfig()` (`packages/core/src/config/config.ts:196-198`, `packages/core/src/config/config.ts:306-315`).
- `schedulerSingleton` can take a `ToolSchedulerFactory` and preserve its existing singleton maps and callback-refresh behavior because only the construction line needs inversion (`packages/core/src/config/schedulerSingleton.ts:78-80`, `packages/core/src/config/schedulerSingleton.ts:201-224`, `packages/core/src/config/schedulerSingleton.ts:265-288`).
- `TaskToolRegistration` descriptor is the correct seam because `ToolRecord` and allow-list logic depend on class/static metadata (`packages/core/src/config/toolRegistryFactory.ts:101-149`, `packages/core/src/config/toolRegistryFactory.ts:247-259`, `packages/core/src/config/toolRegistryFactory.ts:308-309`).
- a2a must wire the same factories because it initializes `Config` (`packages/a2a-server/src/config/config.ts:43-45`, `packages/a2a-server/src/config/config.ts:135-145`), and direct `AgentClient` construction in `a2a-server/src/agent/task.ts` can flip to agents in P03 without needing a core factory (`packages/a2a-server/src/agent/task.ts:145-154`).

The atomic P03 strategy is executable, but it is large. The plan mitigates this with preflight, git-mv, checkpoint commits, generated audit tables, and full-battery verification. The plan should additionally fix the regex and naming issues above to reduce execution errors.

## TDD / verification rigor assessment

P01's TDD guidance is acceptable for a refactor seam: it requires behavioral tests first, forbids mock-theater-only assertions, and demands real history handoff, singleton behavior, and ToolRecord parity (`project-plans/issue1592/plan/01-contracts-inversion.md:46-52`). The plan correctly deviates from full PLAN.md new-feature pseudocode/TDD sequencing because this is a behavior-preserving move of existing code, not new functionality (`project-plans/issue1592/plan/00-overview.md:12-16`).

Verification rigor is generally strong: anti-shim scans, dependency-direction scans, no core-to-agents tests, provider-test migration checks, behavior-preservation diff audits, full test/build/lint/typecheck, smoke, bundle checks, release/CI/Docker/script wiring, and final madge cycle checks are all present (`project-plans/issue1592/plan/03a-code-move-verification.md:7-25`, `project-plans/issue1592/plan/04a-consumer-migration-verification.md:1-6`, `project-plans/issue1592/plan/05-cleanup-final.md:1-7`). Fix the P02 full-battery omission and P03/P03a regex gaps before approving.
