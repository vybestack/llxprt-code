# Plan Review Round 11 — Issue #1592 Extract packages/agents

Verdict: **REVISE**

I reviewed the requested plan files only and ignored `project-plans/issue1592/reviews/`. I also verified the plan against the live codebase with fresh reads/greps rather than relying on the analysis documents.

## Findings

### BLOCKER — P01 is not executable as written: CLI/a2a cannot wire `taskToolRegistration` to the concrete `TaskTool` while it still lives inside core but is not exported by core's public package API

P01 requires the CLI and a2a composition roots to pass all concrete factories/registrations while the concrete classes still live in core: the phase says composition roots including a2a must pass concrete factories via `ConfigParameters`, and explicitly says P01 keeps `a2a-server/src/agent/task.ts` importing `AgentClient` from core until P03 (`project-plans/issue1592/plan/01-contracts-inversion.md:55-60`). The phase summary similarly says P01 wires CLI/a2a/test registration while classes are still in core (`project-plans/issue1592/plan/00-overview.md:37-39`).

That is feasible for `AgentClient` and `CoreToolScheduler` because core's root barrel currently exports them: `packages/core/src/index.ts:73` exports `./core/client.js`, and `packages/core/src/index.ts:82` exports `./core/coreToolScheduler.js`. It is **not** feasible for `TaskTool`: the concrete class is internal today, imported by `toolRegistryFactory` from `../tools/task.js` (`packages/core/src/config/toolRegistryFactory.ts:38`), and core's public barrel around the moved exports does not export `tools/task.js` or `TaskTool` (`packages/core/src/index.ts:72-85`). Core's `package.json` export map also has tool subpaths for shared/base tool modules such as `./tools/tool-registry.js`, `./tools/tools.js`, and related helpers, but no `./tools/task.js` entry (`packages/core/package.json:65-73`).

This matters because a2a imports core as an external package (`packages/a2a-server/src/config/config.ts:12-25`) and constructs/initializes `Config` immediately (`packages/a2a-server/src/config/config.ts:31-45`, `packages/a2a-server/src/config/config.ts:135-145`). P01 removes the core-local `TaskTool` import from `toolRegistryFactory` and requires external composition roots to provide the descriptor (`project-plans/issue1592/plan/01-contracts-inversion.md:30-40`, `project-plans/issue1592/plan/01-contracts-inversion.md:53-60`), but gives them no legal import path for the concrete `TaskTool` until P03 creates the agents package API.

A worker following P01 has bad choices:

- add a new temporary core public export/subpath for `TaskTool`, which the plan does not specify and would create transient public API churn that must be removed later;
- import an unexported core internals subpath from CLI/a2a, which can fail under Node package `exports` and violates the clean-boundary direction of the plan;
- leave `toolRegistryFactory` importing `TaskTool`, which means P01 does not actually sever the class dependency it is supposed to sever;
- or omit a2a `taskToolRegistration`, which the plan correctly says is a behavior regression because `resolveManagers` auto-creates the managers (`packages/core/src/config/toolRegistryFactory.ts:207-226`) and currently registers `TaskTool` in the normal path (`packages/core/src/config/toolRegistryFactory.ts:247-250`).

Required revision: make this phase ordering explicit and executable. For example, either scaffold a legal temporary import path before P01 and explicitly remove it in P03, or split the seam so P01 can keep a core-local default descriptor until P03 atomically flips consumers, or reorder scaffold/entry creation before P01. The plan must state exactly how CLI/a2a obtain the concrete `TaskTool` descriptor during P01 without relying on an unexported core internals import or a forbidden compatibility shim.

### MAJOR — P02 still misses `scripts/check-settings-boundary.js`, an existing provider-aware boundary script that should be audited or explicitly waived

P02's CI/release wiring checklist covers release workflow, sandbox workflow, `scripts/prepare-package.js`, `scripts/build_sandbox.js`, `scripts/version.js`, `Dockerfile`, release-process tests, esbuild, and publish config (`project-plans/issue1592/plan/02-package-scaffold.md:36-45`). It does **not** mention `scripts/check-settings-boundary.js`.

The live code has a boundary script with hard-coded provider/package boundary knowledge: it defines `@vybestack/llxprt-code-providers` as a forbidden settings dependency (`scripts/check-settings-boundary.js:24-30`). A fresh grep also found provider source/package references in that same script at multiple locations, including provider package/source checks (`scripts/check-settings-boundary.js:174`, `scripts/check-settings-boundary.js:194`, `scripts/check-settings-boundary.js:297`, `scripts/check-settings-boundary.js:478`, `scripts/check-settings-boundary.js:508`, `scripts/check-settings-boundary.js:757`).

Because this extraction creates a new package with explicit forbidden directions (`core -X-> agents`, `agents -X-> providers/cli`, and `providers -X-> agents` in `project-plans/issue1592/specification.md:13-25`), this script is a real boundary surface. It may not need functional changes if it is narrowly settings-only, but the plan should require an audit and a documented decision. Otherwise the implementation can miss a CI/boundary precedent while still claiming providers-precedent completeness.

Required revision: add `scripts/check-settings-boundary.js` to P02/P02a or P05/P05a as an explicit audit item: either update it to include agents where appropriate, or document with evidence why the settings-boundary script intentionally remains unchanged.

### MINOR — The P01 verification grep for `TaskTool` is narrower than the actual post-P01 invariant

P01's verification checks only for `from '../tools/task.js'` under `packages/core/src/config` (`project-plans/issue1592/plan/01-contracts-inversion.md:70-76`). The actual dependency boundary after P01 is broader: core config should not import the concrete `TaskTool` via any path form if the seam was successful. The current file uses `import { TaskTool } from '../tools/task.js'` (`packages/core/src/config/toolRegistryFactory.ts:38`), so the specific grep catches today's case, but it would miss an equivalent import via `../../tools/task.js`, a package subpath, a dynamic import, or a renamed import.

Required revision: make the P01a verification use the same multi-form import scan style the plan already requires in P03/P03a, scoped to `toolRegistryFactory.ts` and the concrete moved modules (`TaskTool`, `AgentClient`, `CoreToolScheduler`).

## Verified factual claims and architecture checks

1. **Integration-first is substantially satisfied after P03.** The plan lists real existing consumers and access paths: CLI config/gemini wiring, `autoPromptGenerator`, non-interactive and zed integration, React scheduler hook, a2a task/config, provider test relocation, core config inversion, and release/CI wiring (`project-plans/issue1592/specification.md:102-124`). Apart from the P01 `TaskTool` importability blocker above, the plan cannot be implemented as a useful isolated `packages/agents` island.

2. **The plan correctly identifies old code removal/no-shim requirements.** It says moved files under `packages/core/src/core` and `packages/core/src/agents` must be deleted from core, moved API exports removed from `packages/core/src/index.ts`, and direct construction of `AgentClient`/`CoreToolScheduler`/`TaskTool` inside core removed (`project-plans/issue1592/specification.md:116-120`). P03's anti-shim checks look for moved implementations remaining in core (`project-plans/issue1592/plan/03-code-move.md:60-74`).

3. **`Config` constructor/factory timing is accurately represented in the current plan.** The constructor only calls `applyConfigParams` (`packages/core/src/config/config.ts:103-107`). `AgentClient` is constructed later in `initialize()` (`packages/core/src/config/config.ts:196-198`) and again in `initializeContentGeneratorConfig()` (`packages/core/src/config/config.ts:306-315`). That supports the plan's use-time absence semantics in the integration contract (`project-plans/issue1592/analysis/integration-contract.md:93-99`) and P01 (`project-plans/issue1592/plan/01-contracts-inversion.md:53-60`).

4. **The `ToolSchedulerFactory` inversion is feasible.** `schedulerSingleton` currently imports scheduler result/class types from `../core/coreToolScheduler.js` (`packages/core/src/config/schedulerSingleton.ts:12-17`) and dynamically imports the concrete scheduler before constructing it (`packages/core/src/config/schedulerSingleton.ts:265-288`). The descriptor in `integration-contract.md` preserves those fields (`project-plans/issue1592/analysis/integration-contract.md:58-80`).

## Overall assessment

The plan is much stronger than a naive extraction plan: it is integration-first, it rejects backward-compatibility shims, the target dependency direction is sound, deviations from the literal issue list are mostly justified by real code, and the package layout/build wiring now matches the providers precedent. The remaining P01 `TaskTool` importability problem is execution-blocking because it appears before the atomic P03 move and before agents has a public API. Fix that ordering/import path design, and add the missed boundary-script audit, then the plan should be ready for another review.

Final verdict: **REVISE**
