# Plan Review Round 5: Issue #1592 Extract packages/agents

## Verdict: REVISE

The plan is substantially stronger than a naive package-extraction plan: it is integration-first, identifies concrete consumers, rejects backward-compatibility shims, includes preflight verification, and correctly recognizes the three production construction couplings that must be inverted before the move. I verified the plan against the real codebase with direct file reads and greps rather than trusting the analysis documents.

However, it still needs revision before execution. The main blocker is an internal dependency-gate contradiction: the specification and package-scaffold phase correctly allow `packages/agents` to depend directly on sibling packages such as settings/auth/telemetry/mcp when the moved code imports them, but the P03a workspace-leakage gate fails any sibling package except core and test-utils. The actual move set already imports settings and auth directly, so a correct implementation cannot pass P03a as written without either creating transitive-dependency leakage or rewriting unrelated architecture.

## Findings

### BLOCKER: P03a workspace-leakage gate contradicts the spec and the actual move-set dependencies

P03a requires extracting all package specifiers under `packages/agents` and failing if any workspace package is not on the allowed list, explicitly allowing only core and test-utils dev-only (`plan/03a-code-move-verification.md:20-24`). That gate contradicts the specification, which explicitly permits `packages/agents -> packages/auth, packages/settings, packages/telemetry, packages/mcp (as needed, same as core)` (`specification.md:15-18`) and only forbids providers and CLI (`specification.md:21-24`, `specification.md:129-133`). It also contradicts P02, which instructs deriving direct dependencies from the generated import inventory and allows settings/telemetry/mcp/auth if direct imports exist (`plan/02-package-scaffold.md:23-25`).

The actual moved production files do have direct sibling-package imports. `tools/task.ts`, which the move-map says moves to agents (`analysis/move-map.md`, section D), imports `ProfileManager` from `@vybestack/llxprt-code-settings` (`packages/core/src/tools/task.ts:27-28`). `core/subagentOrchestrator.ts`, also in the move set, imports from `@vybestack/llxprt-code-settings` (`packages/core/src/core/subagentOrchestrator.ts:16`, `packages/core/src/core/subagentOrchestrator.ts:44`). `core/StreamProcessor.ts`, also in the move set, imports `flushRuntimeAuthScope` from `@vybestack/llxprt-code-auth` (`packages/core/src/core/StreamProcessor.ts:27`).

This is not just a wording problem: P03a is a required verification gate before P04 (`plan/04-consumer-migration.md:7-9`). As written, a correct implementation with explicit direct dependencies on settings/auth would fail P03a, while omitting those dependencies would violate P02's no-transitive/workspace-leakage rule (`plan/02-package-scaffold.md:23-25`). Revise P03a line 24 to allow every workspace package permitted by REQ-DEP-001 and actually present in the generated import inventory, while still hard-failing providers, CLI, and the root CLI package.

### MAJOR: Verification commands/gates have inconsistent smoke-test and format commands

The project memory requires the full check-in battery to run `npm run test`, `npm run lint`, `npm run typecheck`, `npm run format`, `npm run build`, and the synthetic smoke test before checking in. The plan's overview says the verification battery uses `npm run format:check` or `npm run format` plus diff check (`plan/00-overview.md:52-58`), while the specification requires `npm run format` as part of full workspace verification (`specification.md:139-143`). Some phase gates omit format entirely: P01's verification commands run typecheck/build/test/lint plus smoke but not format (`plan/01-contracts-inversion.md:71-78`), P03's verification commands also omit format and smoke (`plan/03-code-move.md:53-73`), and P03a says full battery is only typecheck/build/test/lint (`plan/03a-code-move-verification.md:7-10`). P04 later includes format:check and smoke (`plan/04-consumer-migration.md:25-33`), and P05 says full battery plus smoke (`plan/05-cleanup-final.md:26-32`).

This inconsistency matters because P03 is the largest atomic edit and the first phase where the new package contains real code. It should end with the same full verification battery the specification requires, including format and smoke, not a reduced set. Revise every verification phase definition so the phrase "full battery" has one authoritative meaning matching the specification and project memory.

### MAJOR: P03a hard scan can falsely fail legitimate contract-only imports from core/client.js

The plan explicitly allows `core/client.ts` to remain in core as contract-only after the concrete `AgentClient` class moves: P03 task 6 says `core/client.ts` or `clientContract.ts` remains contract-only in core (`plan/03-code-move.md:34-36`), and the specification says core keeps exporting core-owned structural contracts such as `AgentClientContract` (`specification.md:68-78`). But P03a hard scan requires core tests to have no references to `core/client.js` and expects an empty result, while only vaguely allowing "staying-module type paths" (`plan/03a-code-move-verification.md:17-19`).

The current code has multiple core tests and utilities importing `../core/client.js`, including `config/config.test.ts` (`packages/core/src/config/config.test.ts:29`), `utils/checkpointUtils.test.ts` (`packages/core/src/utils/checkpointUtils.test.ts:19`), and `utils/summarizer.test.ts` (`packages/core/src/utils/summarizer.test.ts:9`). After inversion, some of these should be retargeted to the core-owned contract module rather than the agents package, and if that contract module is deliberately `core/client.ts`, the scan will flag correct code. The plan should choose one contract path (`core/clientContract.ts` is less ambiguous) or change the scan to distinguish contract-only imports from moved implementation imports.

### MINOR: Reverse-dependency map understates existing CLI deep imports from core

The reverse-dependency map claims CLI imports everything via the `@vybestack/llxprt-code-core` root except one deep import in `consumer-migration-p13.integration.test.ts` (`analysis/reverse-dependency-map.md`, section 5). The real code has many CLI deep imports from core, including `runtime/settingsRuntimeAdapter.js` in `packages/cli/src/config/profileBootstrap.ts:17`, `packages/cli/src/auth/provider-usage-info.ts:20`, `packages/cli/src/providers/providerManagerInstance.ts:24`, `packages/cli/src/runtime/runtimeContextFactory.ts:36`, and `packages/cli/src/nonInteractiveCli.ts:26`, plus `test-utils/mock-tool.js` in `packages/cli/src/ui/hooks/useToolScheduler.test.ts:51`.

This does not invalidate the extraction because these are not moved modules, but it is a factual error in an analysis artifact. Revise the wording to say there are no CLI deep imports of moved modules except the explicitly relevant contentGenerator test case, rather than saying CLI has only one deep import overall.

### MINOR: P02 verification greps `scripts/prepare-package.js` even though P02 permits documenting no change

P02 task 8 says `scripts/prepare-package.js` currently prepares only core and CLI, and instructs the implementer to determine whether agents needs handling; if not, document why (`plan/02-package-scaffold.md:32-36`). But P02's verification command unconditionally greps for `agents` in `scripts/prepare-package.js` (`plan/02-package-scaffold.md:44-52`). If the correct conclusion is "no prepare-package change needed," that grep will fail or produce misleading output. Make the verification conditional: either `scripts/prepare-package.js` contains agents handling, or the P02 completion marker cites evidence that no handling is needed.

## Integration-first and isolation assessment

The plan passes the integration-first check. It lists existing code that will use the extracted package, including CLI config/gemini bootstrap, autoPromptGenerator, non-interactive support, zed integration, React scheduler hook, a2a task/config, provider test relocation, core config inversion, and release/CI wiring (`specification.md:99-112`). It also lists old code to remove: moved files under core/agents, moved exports from `packages/core/src/index.ts`, and direct construction of moved classes inside core (`specification.md:113-118`). P03 explicitly combines code move and consumer migration in one atomic phase because shims are forbidden (`plan/03-code-move.md:11-14`, `plan/03-code-move.md:43-50`). Therefore this cannot be built as an isolated `packages/agents` island without updating existing consumers.

The old-code removal plan is credible: P03 verifies no moved implementation remains in core (`plan/03-code-move.md:53-60`), P03a checks move-map compliance and export cleanup (`plan/03a-code-move-verification.md:9-18`), and P05 prunes moved export-map entries and dead exports (`plan/05-cleanup-final.md:16-28`).

## Deviation assessment

The deviations from the issue's literal file list are mostly correct and justified:

- `core/contentGenerator.ts` should stay because it is a runtime/model-I/O contract used by core config/runtime/code-assist/test-utils and providers. The plan documents this (`specification.md:34-39`), and provider/runtime evidence supports it.
- `core/prompts.ts` should stay because production provider implementations import it; moving it would create providers -> agents coupling.
- `core/tokenLimits.ts` should stay because core runtime and CLI consumers use it independently of chat-loop implementation.
- `core/loggingContentGenerator.ts` does not exist in the current tree.
- `core/logger.ts` staying is plausible: it imports only settings storage (`packages/core/src/core/logger.ts:12`) and is not clearly agent runtime machinery.
- `core/googleGenAIWrapper.ts` staying is plausible because the move-map says it is used by staying `contentGenerator.ts`; verify this again in P00a before implementation.
- `coreToolHookTriggers.ts` and `lifecycleHookTriggers.ts` staying are justified by staying hook exports/tests and staying-module dependencies, as cited above.
- `ListSubagentsTool` and `SubagentManager` staying is sound because they represent subagent configuration/registry rather than runtime execution; `toolRegistryFactory.ts` imports `ListSubagentsTool` separately from `TaskTool` at `packages/core/src/config/toolRegistryFactory.ts:38-39`, and only `TaskTool` constructs subagent runtime machinery.

## Dependency-direction assessment

The intended architecture is sound: core must not import agents; agents must not import providers or CLI; providers must not depend on agents; consumers that need concrete agent runtime import agents directly. The plan includes relevant scans in overview/P03/P03a/P04 (`plan/00-overview.md:60-72`, `plan/03-code-move.md:61-70`, `plan/03a-code-move-verification.md:20-25`, `plan/04-consumer-migration.md:31-33`).

The one required revision is to align the allowed sibling dependency list with the specification and actual code. Agents should be allowed to directly depend on settings/auth/telemetry/mcp if and only if the generated import inventory proves direct imports. Providers and CLI must remain hard-fail dependencies/imports in all import forms and all dependency sections.

## Phase-ordering and construction-inversion feasibility

The phase ordering is executable after the dependency-gate fix. P01 inverts construction while classes still live in core, P02 creates the empty package and release/build wiring, P03 atomically moves code and flips consumers, P04 audits/hardens consumers, and P05 prunes surfaces. This respects the no-shim constraint and keeps phase endpoints green.

The construction-inversion design is feasible and behavior-preserving:

- `AgentClientFactory` must be a constructor parameter or otherwise present before `initialize()` because `Config.initialize()` constructs `AgentClient` at `packages/core/src/config/config.ts:196-198`, and `initializeContentGeneratorConfig()` constructs another at `packages/core/src/config/config.ts:314-315`.
- Scheduler factory inversion is feasible because the singleton already centralizes creation in `createNewScheduler()` (`packages/core/src/config/schedulerSingleton.ts:265-310`); replacing only the dynamic import/new expression should preserve registry/ref-count/callback behavior.
- TaskTool descriptor inversion is preferable to a bare factory because `toolRegistryFactory.ts` relies on class constructor metadata for ToolRecord and allow-list/exclude behavior (`packages/core/src/config/toolRegistryFactory.ts:101-149`, `packages/core/src/config/toolRegistryFactory.ts:247-259`, `packages/core/src/config/toolRegistryFactory.ts:304-310`). The plan correctly requires parity tests for registered, missing-manager, allow-list, and exclude scenarios (`analysis/integration-contract.md`, TaskToolRegistration section; `plan/03a-code-move-verification.md:25`).
