- All `RuntimeTokenizer.test.ts`, `RuntimeProvider.test.ts`, `RuntimeContentGeneratorFactory.test.ts`, `structural-contracts.test.ts` tests would fail at import (type imports would not resolve)
- `MissingRuntimeProviderError.test.ts` would fail (class import would not resolve)
- `toolIdNormalization-contract.test.ts` would fail (function import from core path would not resolve)
- `boundary-guards.test.ts` path-based scan tests would find 0 contract files and fail

This confirms the tests are not structure-only — they exercise real behavior of the P03 stubs through structural typing and functional verification.

## Tests That Will Fail Until P05 Implementation

None of the Phase 04 tests are expected to initially fail due to missing P05 implementation. All tests pass against the P03 stubs because:

1. **Structural contract tests** verify that the contract types are structurally compatible with provider shapes. These pass with P03 stubs because the contracts are type-only (interfaces/type aliases).
2. **MissingRuntimeProviderError tests** verify the error class behavior. These pass because P03 created the concrete error class.
3. **toolIdNormalization tests** verify the moved utility function. These pass because P03 moved the complete implementation.
4. **Boundary guard tests** verify that contract files don't import providers. These pass because P03 stubs were created without provider imports.

Tests that would initially fail until P05 implementation would be:
- Tests verifying that consumer files (HistoryService, contentGenerator, configTypes) use the new contracts instead of provider imports. These are P05a verification tests, not P04 scope.
- Tests verifying that the `runtime/index.ts` barrel re-exports all contracts correctly. This passes now because P03 added the barrel exports.
