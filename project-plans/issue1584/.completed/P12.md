## Summary

P12 migrated CLI consumer wiring and tests away from provider-owned exports in `@vybestack/llxprt-code-core` and toward `@vybestack/llxprt-code-providers`, while keeping core-owned runtime/config/settings/profile APIs imported from core. ## P12a Remediation

P12a initially reported remaining imports of a core `ProviderManager` alias and a core runtime-contract alias that looked like a provider compatibility shim. Remediation removed the `ProviderManager = RuntimeProviderManager` alias from core runtime contracts and changed CLI code to import the concrete provider manager type from `@vybestack/llxprt-code-providers` where concrete provider-manager behavior is required, or to use a local alias to core-owned `RuntimeProviderManager` where only structural runtime access is required.

A follow-up untracked-file inspection showed no generated `.js`, `.d.ts`, `.js.map`, or `.tsbuildinfo` artifacts in core/providers source paths. The remaining untracked core/providers files are intentional new source/test files for P03-P11/P12, not generated build artifacts.
